# 可观测性、安全与部署设计

## 1. 模块定位

本模块覆盖三类横切能力：可观测性、安全治理和基础部署。它们不直接决定某一步怎么执行，但决定平台是否稳定、可审计、可扩展、可上线。

> 当前实现状态：当前仓库仍以单体式 MVP 为主，已经具备最基础的异常处理、后台异步执行和步骤事件流，但完整的指标、Tracing、租户隔离、鉴权、审批安全链路和平台化部署能力尚未落地。

## 2. 可观测性设计

### 2.1 Metrics

目标建议至少监控：

- Run 数量
- 完成率
- 中断率
- 平均轮次
- 工具调用次数
- 模型调用耗时
- 工具耗时
- 审批等待时长
- 单 Run 成本

当前实现状态：

- 当前尚未落地独立 Metrics 体系
- 当前最接近观测数据的能力主要体现在：
  - `AgentStep.DurationMs`
  - `AgentRun` / `AgentStep` 持久化记录
  - SSE 运行事件流
- 单 Run 成本、模型耗时聚合、审批等待时长等指标尚未形成

### 2.2 Logging

目标建议分层日志：

- API Log
- Runtime Log
- Model Log
- Tool Log
- Policy Log

当前实现状态：

- 当前已具备基础 Runtime 错误日志，例如 `AgentRunWorker` 处理失败日志
- 当前尚未形成清晰分层的结构化日志体系
- Model / Tool / Policy 级别的独立日志层仍未完整落地

### 2.3 Tracing

目标建议为以下对象分配 trace/span：

- AgentRun
- ModelCall
- ToolExecution
- Retrieval
- Approval
- Handoff

当前实现状态：

- 尚未落地分布式 Tracing

## 3. 安全与治理设计

### 3.1 鉴权

目标必须支持：

- 用户身份鉴权
- 服务身份鉴权
- 工具级权限控制

当前实现状态：

- 当前尚未落地完整身份鉴权
- 工具级别仅实现了基于 `AllowedTools` 的 allowlist 控制

### 3.2 数据隔离

目标必须支持：

- Tenant 隔离
- User 隔离
- 环境隔离

当前实现状态：

- `AgentRun` 模型中已有 `TenantId`、`UserId`、`SessionId` 等字段
- 但当前尚未形成真正的租户 / 用户隔离治理逻辑

### 3.3 Prompt 安全

目标建议支持：

- Prompt 模板版本管理
- Prompt 注入防护
- 输入输出审查

当前实现状态：

- 当前已支持 `system_prompt_template` 与 AgentDefinitionVersion 绑定
- 当前尚未落地系统化的 Prompt 注入防护与输入输出审查链路

### 3.4 工具安全

目标建议支持：

- allowlist
- denylist
- 参数黑白名单
- 敏感工具审批

当前实现状态：

- 当前已落地：
  - allowlist（`AllowedTools`）
- 当前尚未落地：
  - denylist
  - 参数黑白名单
  - 敏感工具审批

### 3.5 审计

目标关键动作必须可追踪：

- 谁发起了请求
- 调用了哪个 Agent
- 执行了哪些工具
- 谁审批了写操作

当前实现状态：

- 当前已能追踪：
  - Run 与 Step
  - 工具执行步骤
  - 基础审计字段（creator / last modifier）
- 当前尚未落地完整用户身份审计和审批审计链路

### 3.6 策略求值顺序

目标建议固定如下顺序：

1. 身份鉴权
2. 租户和环境隔离
3. Agent allowlist
4. Tool allowlist/denylist
5. 参数级校验
6. 审批策略
7. 执行

当前实现状态：

- 当前最明确落地的是：
  1. AgentDefinition 加载
  2. Tool allowlist（`AllowedTools`）
  3. 工具执行
- 身份鉴权、租户隔离、参数级策略校验、审批策略尚未完整进入主链路

## 4. 基础设施建议

目标基础能力包括：

- DB
- Cache
- Queue
- Object Storage
- Vector Store
- Secret Manager

当前实现状态：

- 当前稳定使用的基础设施主要是：
  - DB（PostgreSQL）
  - 进程内异步队列（`Channel<T>`）
- 当前尚未引入：
  - 独立 Cache
  - Object Storage
  - Vector Store
  - Secret Manager
  - 进程外消息队列

## 5. 部署形态建议

### 5.1 MVP 形态

单体服务即可：

- API
- Runtime
- Tool Executor
- DB
- Cache

当前实现状态：

- 当前正是单体式 MVP 形态
- API、Runtime、Tool Executor 都运行在同一进程中
- 当前尚未依赖外部 Cache

### 5.2 平台化形态

目标建议拆分：

- API Service
- Runtime Service
- Tool Service
- Retrieval Service
- Approval Service
- Admin Service

当前实现状态：

- 尚未拆分

### 5.3 异步化建议

目标建议：

- Run 创建同步返回
- 实际执行异步推进
- 前端通过事件流或轮询获取结果

当前实现状态：

- 已落地
- 当前通过：
  - `POST /agent-runs` 同步创建
  - `Channel + BackgroundService` 异步执行
  - `GET /agent-runs/{runId}/stream` SSE 推送结果

## 7. 风险控制点

目标设计：

- 没有可观测性就无法稳定运营长任务 Agent
- 没有隔离和审批就无法开放高权限工具
- 没有异步与恢复骨架就无法支撑真实业务负载
- 过早引入复杂多 Agent 会拖慢第一阶段落地

当前实现状态：

- 当前已经验证了“异步与恢复骨架”对 MVP 的必要性
- 当前最大的现实风险是：
  - 缺少完整可观测性
  - 缺少租户 / 身份安全边界
  - 工具高权限治理仍未落地

## 8. 验收重点

目标验收：

- 关键运行链路具备 Metrics、Logs、Tracing
- 关键动作具备身份、租户、工具三级控制
- MVP 到平台化拆分路径清晰
- 演进阶段与能力引入顺序符合源设计文档约束

当前实现状态：

- 当前更适合的验收重点是：
  - 单体 MVP 主链路可运行
  - 异步执行与 SSE 推送可用
  - Tool allowlist 和基础异常处理可工作
- Metrics / Tracing / 身份租户安全链路 / 平台化拆分仍属于后续目标态
