# 可观测性、安全与部署设计

## 1. 模块定位

本模块覆盖三类横切能力：可观测性、安全治理和基础部署。它们不直接决定某一步怎么执行，但决定平台是否稳定、可审计、可扩展、可上线。

> 当前实现状态：当前仓库仍以单体式 MVP 为主，已经具备最基础的异常处理、后台异步执行和步骤事件流，但完整的指标、Tracing、租户隔离、鉴权、审批安全链路和平台化部署能力尚未落地。

## 2. 可观测性设计

### 2.1 Metrics

目标建议至少监控：

- Run 数量
- 完成率
- 中断率
- 平均轮次
- 工具调用次数
- 模型调用耗时
- 工具耗时
- 审批等待时长
- 单 Run 成本

当前实现状态：

- 当前已补上最小 `System.Diagnostics.Metrics` 指标接入，覆盖：
  - Run 创建计数
  - Run 终态计数（`Completed` / `Failed` / `Cancelled` / `TimedOut`）
  - 模型调用次数、耗时、token 聚合与单次 cost
  - 工具执行次数与耗时
  - 检索调用次数、耗时、候选数与选中 chunk 数
  - 审批等待次数、等待时长与超时次数
- 当前仍未接入独立 exporter / dashboard，也尚未形成完整平台级 Metrics 管道
- 平均轮次、跨实例聚合视角与更细粒度业务指标仍待继续补齐

### 2.2 Logging

目标建议分层日志：

- API Log
- Runtime Log
- Model Log
- Tool Log
- Policy Log

当前实现状态：

- 当前已具备基础 Runtime 错误日志，例如 `AgentRunWorker` 处理失败日志
- 当前已开始补最小分层结构化日志：
  - `BestAgentRequestLoggingMiddleware` 会记录 API 请求方法、路径、状态码、耗时与最小身份/scope 信息
  - `OpenAiCompatibleModelGateway` 会记录模型调用开始、成功完成和失败日志
  - `ToolExecutor` 会记录工具执行开始、完成状态和失败日志
  - `DefaultApprovalAuthorizer` 会记录审批授权通过/拒绝的最小 policy decision 日志
- 更完整 API / Policy 日志规范与更多细分层仍未完整落地

### 2.3 Tracing

目标建议为以下对象分配 trace/span：

- AgentRun
- ModelCall
- ToolExecution
- Retrieval
- Approval
- Handoff

当前实现状态：

- 当前已补上最小 `ActivitySource` Tracing 挂点，开始覆盖：
  - `AgentRun` 后台消息处理 span
  - `ModelCall` span
  - `ToolExecution` span
  - `Retrieval` span
  - `Approval` 审批/人工完成/超时 span
  - `Handoff` 子 Run 派发 span
- 当前仍未接入 OpenTelemetry exporter、跨进程 trace 传播和完整平台级 trace pipeline

## 3. 安全与治理设计

### 3.1 鉴权

目标必须支持：

- 用户身份鉴权
- 服务身份鉴权
- 工具级权限控制

当前实现状态：

- 当前已开始落地最小身份鉴权中间件：
  - ASP.NET Core 认证管道已接入配置化 `Bearer` token 用户/服务身份
  - 用户身份可写入 `user_id` / `tenant_id` / `session_id` / `role` claims
  - 服务身份可写入 `service_id` / `role` claims
  - 当请求显式携带 `Authorization` 但凭据无效时，会在进入业务控制器前返回 `401`
  - 当 `Authentication:RequireAuthenticatedRunAccess=true` 时，交互式 Run API 会进一步要求已认证身份；外部 tool/approval complete 回调继续沿用 HMAC 校验
  - 当 `Authentication:RequireAuthenticatedManagementAccess=true` 时，`AgentDefinitionsController` / `ToolDefinitionsController` 管理接口也会进一步要求已认证身份；若进一步配置 `Authentication:ManagementAllowedRoles`，还会执行最小角色校验
  - 当配置 `HumanTakeover:AllowedRoles` 时，`request-human` / `complete-human` 也会在应用层执行最小人工处理人角色校验
- 工具级别仅实现了基于 `AllowedTools` 的 allowlist 控制
- `AgentRunsController` 当前已开始最小消费身份上下文：若请求带有 `tenant/user/session` claims，则创建 Run 时会优先继承这些字段；在尚未接入正式鉴权时，也可显式传入 `X-BestAgent-Tenant-Id` / `X-BestAgent-User-Id` / `X-BestAgent-Session-Id`。在存在这些 scope 时，Run 查询、恢复、取消、审批、人机协同、外部 tool/approval complete 回调与 SSE stream 入口会校验 Run 是否仍落在相同 scope 内

### 3.2 数据隔离

目标必须支持：

- Tenant 隔离
- User 隔离
- 环境隔离

当前实现状态：

- `AgentRun` 模型中已有 `TenantId`、`UserId`、`SessionId` 等字段
- 当前已开始具备最小 request-side 租户 / 用户 / session 边界治理：带认证上下文的 Run 相关 API 会按相同 scope 校验访问
- 但当前尚未形成真正完整的租户 / 用户隔离体系，匿名请求兼容路径之外的强制鉴权策略、细粒度授权策略与跨实例边界治理仍未落地

### 3.3 Prompt 安全

目标建议支持：

- Prompt 模板版本管理
- Prompt 注入防护
- 输入输出审查

当前实现状态：

- 当前已支持 `system_prompt_template` 与 AgentDefinitionVersion 绑定
- 当前尚未落地系统化的 Prompt 注入防护与输入输出审查链路

### 3.4 工具安全

目标建议支持：

- allowlist
- denylist
- 参数黑白名单
- 敏感工具审批

当前实现状态：

- 当前已落地：
  - allowlist（`AllowedTools`）
  - denylist（`DeniedTools`），Runtime 会在工具执行前先阻断被显式拒绝的工具
  - 参数黑白名单（tool `ParameterPolicy.allowedPaths/deniedPaths`）
- 当前尚未完整落地：
  - 更细粒度敏感工具审批策略
  - 更复杂 deny/allow 组合与租户级工具权限治理

### 3.5 审计

目标关键动作必须可追踪：

- 谁发起了请求
- 调用了哪个 Agent
- 执行了哪些工具
- 谁审批了写操作

当前实现状态：

- 当前已能追踪：
  - Run 与 Step
  - 工具执行步骤
  - 基础审计字段（creator / last modifier）
- 当前尚未落地完整用户身份审计和审批审计链路

### 3.6 策略求值顺序

目标建议固定如下顺序：

1. 身份鉴权
2. 租户和环境隔离
3. Agent allowlist
4. Tool allowlist/denylist
5. 参数级校验
6. 审批策略
7. 执行

当前实现状态：

- 当前最明确落地的是：
  1. 身份鉴权中间件（最小 Bearer 用户 / 服务身份）
  2. AgentDefinition 加载
  3. Tool allowlist / denylist
  4. 已认证请求下的最小 tenant/user/session scope 校验
  5. 参数级策略校验
  6. 审批策略 MVP
  7. 工具执行
- 但完整租户隔离、强制授权模型与更细粒度策略仍未完整进入主链路

## 5. 部署形态建议

### 5.1 MVP 形态

单体服务即可：

- API
- Runtime
- Tool Executor
- DB
- Cache

当前实现状态：

- 当前正是单体式 MVP 形态
- API、Runtime、Tool Executor 都运行在同一进程中
- 当前尚未依赖外部 Cache

### 5.2 平台化形态

目标建议拆分：

- API Service
- Runtime Service
- Tool Service
- Retrieval Service
- Approval Service
- Admin Service

当前实现状态：

- 尚未拆分

### 5.3 异步化建议

目标建议：

- Run 创建同步返回
- 实际执行异步推进
- 前端通过事件流或轮询获取结果

当前实现状态：

- 已落地
- 当前通过：
  - `POST /agent-runs` 同步创建
  - `Channel + BackgroundService` 异步执行
  - `GET /agent-runs/{runId}/stream` SSE 推送结果

## 7. 风险控制点

目标设计：

- 没有可观测性就无法稳定运营长任务 Agent
- 没有隔离和审批就无法开放高权限工具
- 没有异步与恢复骨架就无法支撑真实业务负载
- 过早引入复杂多 Agent 会拖慢第一阶段落地

当前实现状态：

- 当前已经验证了“异步与恢复骨架”对 MVP 的必要性
- 当前最大的现实风险是：
  - 缺少完整可观测性
  - 缺少租户 / 身份安全边界
  - 工具高权限治理仍未落地

## 8. 验收重点

目标验收：

- 关键运行链路具备 Metrics、Logs、Tracing
- 关键动作具备身份、租户、工具三级控制
- MVP 到平台化拆分路径清晰
- 演进阶段与能力引入顺序符合源设计文档约束

当前实现状态：

- 当前更适合的验收重点是：
  - 单体 MVP 主链路可运行
  - 异步执行与 SSE 推送可用
  - Tool allowlist 和基础异常处理可工作
- Metrics / Tracing / 身份租户安全链路 / 平台化拆分仍属于后续目标态
