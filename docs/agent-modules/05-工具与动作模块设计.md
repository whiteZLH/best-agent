# 工具与动作模块设计

## 1. 模块定位

工具与动作模块负责把 Agent 的结构化决策转成可执行动作。它承载工具定义、可见性解析、参数校验、权限控制、实际执行、超时重试、幂等与补偿。

## 2. 设计目标

- 工具注册与执行解耦
- 工具调用全程结构化、可审计
- 支持同步与异步模式
- 对副作用工具建立分级控制

## 3. 模块组成

### 3.1 Tool Registry

负责管理所有 `ToolDefinition`。

### 3.2 Tool Resolver

根据 AgentDefinition、租户、环境、权限决定本次可见工具。

### 3.3 Tool Validator

负责：

- JSON Schema 校验
- 类型校验
- 安全校验

### 3.4 Tool Executor

负责：

- 实际执行工具
- 超时控制
- 重试控制
- 幂等控制

### 3.5 Tool Policy

负责副作用级别分级和审批联动。

## 4. 核心对象

### 4.1 ToolDefinition

建议字段：

- `tool_name`
- `description`
- `input_schema`
- `output_schema`
- `side_effect_level`
- `timeout_ms`
- `retry_policy`
- `auth_policy`
- `idempotency_policy`
- `async_supported`
- `consistency_mode`
- `compensation_policy`
- `enabled`

### 4.2 ToolInvocation

建议字段：

- `invocation_id`
- `run_id`
- `step_id`
- `tool_name`
- `status`
- `input_payload`
- `output_payload`
- `error_payload`
- `idempotency_key`
- `callback_token`
- `executor_node`
- `started_at`
- `ended_at`

状态建议：

- `Pending`
- `Running`
- `Succeeded`
- `Failed`
- `Expired`
- `Cancelled`

## 5. 副作用分级

- `read_only`
- `internal_write`
- `external_write`
- `destructive`

分级越高，越应要求审批、显式确认或人工接管。

## 6. 调用模式

### 6.1 同步模式

- Runtime 调用后在当前执行片段拿到结果
- 适用于短耗时查询和确定性操作

### 6.2 异步模式

- 先创建 `ToolInvocation`
- Run 切换为 `WaitingTool`
- 等待回调或轮询结果

异步工具必须提供：

- `callback_token`
- 过期时间
- 幂等键

## 7. 标准结果结构

所有工具结果建议归一化为：

```json
{
  "status": "succeeded",
  "data": {},
  "error": null,
  "meta": {
    "durationMs": 320,
    "source": "order-service",
    "cached": false
  }
}
```

## 8. 幂等与补偿

- 每次调用都生成 `idempotency_key`
- 写操作工具至少满足“至少一次回调、至多一次生效”
- 不可逆副作用工具必须声明 `compensation_policy`
- 若工具侧无法保证幂等，Runtime 侧必须前置去重

## 9. 安全控制点

- 工具只允许从 allowlist 中解析
- 参数必须通过 schema 与黑白名单校验
- 敏感参数在日志中应脱敏
- destructive 工具默认要求审批

## 10. 验收重点

- 同一工具调用重放不会重复生效
- 同步与异步模式都能回写标准结果
- 工具错误会被结构化记录并可被 Reflector 消费
- 高风险工具调用会触发正确的审批链路
