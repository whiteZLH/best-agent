# 工具与动作模块设计

## 1. 模块定位

工具与动作模块负责把 Agent 的结构化决策转成可执行动作。它承载工具定义、可见性解析、参数校验、权限控制、实际执行、超时重试、幂等与补偿。

> 当前实现状态：已落地统一工具执行入口 `IToolExecutor`，并通过 `AgentRunLoop -> ToolExecutor` 接入主运行时；当前已加入最小运行时输入 / 输出 schema 校验，并开始落地最小 Runtime 前置去重，但完整的工具解析、校验、策略编排仍未拆分成独立子模块。

## 2. 设计目标

- 工具注册与执行解耦
- 工具调用全程结构化、可审计
- 支持同步与异步模式
- 对副作用工具建立分级控制

> 当前实现状态：
> - “工具定义”和“工具执行”已经初步解耦：`ToolDefinition` 负责持久化定义，`ToolExecutor` / `HttpToolInvoker` 负责执行。
> - 当前更接近 **MVP 过渡形态**，不是最终平台态：数据库定义已成为主要事实来源，但本地工具与 HTTP 工具仍是并行模型。

## 3. 模块组成

### 3.1 Tool Registry

目标职责：管理系统内可用工具的执行绑定与本地实现。

当前实现状态：

- 当前 `ToolRegistry` **不负责管理所有 `ToolDefinition`**
- 它只负责管理进程内本地 handler 映射
- 当前内置了两个本地 handler：`echo_context`、`async_task`
- 在当前实现中，它主要承担兼容回退角色，而不是主工具目录

### 3.2 Tool Resolver

目标职责：根据 `AgentDefinition`、租户、环境、权限决定本次可见工具，并解析本次调用应绑定到哪个执行后端。

当前实现状态：

- 当前已经补上独立 `ToolResolver` 组件
- 现阶段工具可用性校验主要体现在 `AgentRunLoop` 对 `AllowedTools` 的检查
- `ToolResolver` 负责把当前 DB-first + 本地 handler 兼容回退规则从 `ToolExecutor` 中拆出，语义为：
  1. 先读取 `ToolDefinition`
  2. 若定义禁用则拒绝执行
  3. 若定义存在显式 `ExecutionKind + ExecutionBinding`，优先按 persisted binding 解析；当前 persisted `ExecutionBinding` 已开始采用最小版本化文档结构
  4. 对仍停留在 legacy webhook flat 字段（`EndpointUrl` / `HttpMethod` / `AuthHeaders`）的旧定义，当前主要通过启动初始化阶段自动回填 persisted webhook binding 再进入主路径
  5. 对新的显式 binding 写入请求，当前已开始收紧 legacy webhook flat 字段兼容：`webhook` binding 下若同时提供 legacy flat 字段，则其值必须与 binding 内载荷一致；`local_handler` / `inline_result` binding 下则要求这些字段显式省略
  6. 对旧的扁平 `ExecutionBinding` JSON 形状，当前解析器仍保留最小兼容读取；若 binding 文档版本超出当前支持范围，则会显式拒绝解析
  7. 若定义存在但既无 persisted binding，启动期也无法从 legacy webhook flat 字段回填，则不再隐式回退到本地 handler，而是直接报错
  8. 当前实现已经要求所有可执行工具必须先有 `ToolDefinition`；本地 handler 只作为显式 `local_handler` binding 的执行后端，不再支持“无定义但已注册即可执行”的路径
- 当前 `ToolResolver` 输出的是**内部执行解析结果**（如 local/webhook/inline），而不是已经持久化的通用 `execution kind` 模型；但 persisted binding 本身已开始具备最小版本化语义
- 启动初始化当前已开始承担一部分过渡治理：会为旧的 webhook 工具定义自动补齐 persisted `ExecutionKind + ExecutionBinding`，也会按 persisted binding 修复漂移的 legacy flat 字段，并清理非 webhook 工具上残留的 legacy webhook flat 字段，尽量减少 runtime 解析 legacy flat 字段的需求

### 3.3 Tool Validator

目标职责：

- JSON Schema 校验
- 类型校验
- 安全校验

当前实现状态：

- 当前已实现 `ToolDefinition` 配置阶段的基础 JSON 校验与最小策略规范化：
  - `InputSchema`
  - `OutputSchema`
  - `AuthHeaders`
  - `ParameterPolicy`（支持最小 `allowedPaths` / `deniedPaths` JSON 对象）
  - `RetryPolicy`（支持 legacy `retry-once` / `retry-twice` 归一化为结构化 JSON）
  - `ConsistencyMode`（当前收紧为 `none` / `eventual` / `strong` 并统一按小写持久化）
  - `SideEffectLevel`（当前收紧为 `read_only` / `internal_write` / `external_write` / `destructive` 并统一按小写持久化）
- 当前已形成最小运行时输入校验组件：`JsonSchemaToolInputValidator`
- 当前已形成最小运行时输出校验组件：`JsonSchemaToolOutputValidator`
- 当前执行前会按 `ToolDefinition.InputSchema` 校验、同步完成后会按 `ToolDefinition.OutputSchema` 校验同一组常用 JSON Schema 子集：
  - `type`
  - `required`
  - `properties`
  - `enum`
  - `additionalProperties`（布尔与对象形态）
- 当前已额外支持一组常用约束：
  - 数组：`items`、`prefixItems`、legacy `items: []` 元组写法、`additionalItems`、`minItems`、`maxItems`、`contains`、`minContains`、`maxContains`、`uniqueItems`
  - 字符串：`minLength`、`maxLength`
  - 数值：`minimum`、`maximum`、`exclusiveMinimum`、`exclusiveMaximum`、`multipleOf`
  - 对象：`minProperties`、`maxProperties`、`propertyNames`、`patternProperties`、`dependentRequired`、`dependentSchemas`
  - 字符串模式与常量：`pattern`、`const`
  - 最小组合关键字：`allOf`、`anyOf`、`oneOf`、`not`、`if/then/else`
  - 最小格式：`email`、`uri`、`date-time`、`date`、`uuid`
- 输出校验当前已覆盖两段主路径：
  - 同步完成结果：在 `ToolExecutor` 返回后立即校验
  - 异步恢复 / 人工替代工具结果：在 `AgentRunWorker` 恢复继续 loop 前做最小后置校验
- 当前执行前也已开始接入最小参数级策略校验：若工具声明 `ParameterPolicy.allowedPaths` / `deniedPaths`，`ToolExecutor` 会在 schema 校验通过后继续按输入路径级规则拦截未放行或显式禁用的参数
- 当前尚未支持完整 JSON Schema 规范、更丰富格式族以及更完整的高级关键字组合

### 3.4 Tool Executor

目标职责：

- 实际执行工具
- 超时控制
- 重试控制
- 幂等控制

当前实现状态：

- 当前已实现统一执行入口 `ToolExecutor`
- `ToolExecutor` 仍采用 **DB-first** 语义，但已不再自己承载全部绑定判断逻辑
- 当前由 `ToolResolver` 先解析执行绑定，再由 `ToolExecutor` 分发到：
  - `HttpToolInvoker`
  - 本地 handler delegate
  - `inline_result` 固定结果返回
- 当前若命中 HTTP webhook 工具，则调用 `HttpToolInvoker`
- `HttpToolInvoker` 当前已覆盖：
  - 超时控制
  - HTTP webhook 最小重试策略
  - AuthHeaders 透传
  - `Idempotency-Key` 最小透传
  - 输入 JSON 透传
  - webhook 响应解析
  - pending / waitToken 协议
  - 最小标准结果字段映射（`status` / `error` / `meta`）
- 当前已落地最小 `IdempotencyPolicy` 执行：
  - `idempotent` 会为 webhook 请求生成并透传 `Idempotency-Key`
  - `non-idempotent` / `disabled` 不会透传幂等键
  - 也兼容最小 JSON 对象策略：`{"enabled":true|false}`
- 当前已开始落地最小 Runtime 前置去重：
  - 同一 run 内，若工具启用了 `IdempotencyPolicy`
  - 若出现“同工具名 + 同输入”的已完成调用记录，Runtime 会直接复用已有结果而不再次执行工具
  - 若出现“同工具名 + 同输入”的 pending 异步调用，Runtime 会直接复用原 `waitToken` / `invocationId` 而不再次发起工具调用
  - 已完成结果复用场景会额外落一条 `mode = reused` 的 `ToolInvocation` 审计记录
- 当前已开始落地最小 `CompensationPolicy` 执行：
  - 当 `CompensationPolicy={"mode":"manual"}` 时，工具执行抛错、工具返回 `failed/error`，以及异步恢复 / 人工替代结果命中输出 schema 校验失败，运行时会自动进入 `WaitingHuman`
  - 当前实现复用既有 `human_wait` 机制保留来源 `stepId`、`invocationId`、工具名、工具输入和结构化失败载荷，等待人工补偿
  - 更丰富的自动补偿动作编排、外部补偿执行器和多阶段补偿策略仍未落地
- 当前 `RetryPolicy` 已支持结构化 JSON 策略和 legacy 字符串兼容：
  - `{"maxAttempts":3,"delayMs":10}`
  - `"retry-once"`
  - `"retry-twice"`
- 当前重试只覆盖 HTTP webhook 的超时、网络异常、`408`、`429` 和 `5xx` 响应；不会重试 `4xx` 参数/权限类错误
- `ToolExecutor` 当前会在真正执行前做输入 schema 校验，并在同步完成且未失败的结果返回后做输出 schema 校验；异步恢复和人工替代工具结果的后置输出校验当前由 `AgentRunWorker` 补齐
- `ToolDefinition` API 当前也已开始支持结构化 `Policies.Parameter` 写侧与 `ParameterPolicy` flat 字段并存输入；两者并存时要求语义一致

### 3.5 Tool Policy

目标职责：负责副作用级别分级和审批联动。

当前实现状态：

- `ToolDefinition` 已持久化 `side_effect_level`
- 当前尚未形成独立 `ToolPolicy` 组件
- 当前副作用等级已接入可配置审批策略 MVP，可按 `Approval:Policy` 决定哪些等级触发审批、哪些等级要求审批角色；默认审批触发等级当前已包含 `internal_write`、`external_write` 与 `destructive`

## 4. 核心对象

### 4.1 ToolDefinition

目标字段：

- `tool_name`
- `description`
- `input_schema`
- `output_schema`
- `side_effect_level`
- `timeout_ms`
- `retry_policy`
- `auth_policy`
- `idempotency_policy`
- `async_supported`
- `consistency_mode`
- `compensation_policy`
- `enabled`

当前实现状态：

- 上述核心字段大多已经落地
- 当前实现额外包含 webhook 执行相关字段：
  - `endpoint_url`
  - `http_method`
  - `auth_headers`
- 当前 persisted `ExecutionBinding` 已开始按 `version + type + payload` 文档结构落库；当前已用于承载 webhook / local handler / `inline_result` 绑定，并为后续更多 executor type 预留扩展位
- 查询读侧当前也会对尚未归一的纯 legacy webhook 记录即时合成结构化 `Execution` 视图，避免 API 返回层只能暴露旧 flat 字段
- 查询读侧当前也会让兼容保留的 flat `ExecutionBinding` 与结构化 `Execution` 视图保持一致的最小脱敏语义：webhook binding 中的认证头与 `inline_result` binding 中的敏感运行时字段都会被脱敏
- 查询读侧当前也已开始补充最小结构化 `Policies` 视图，显式展开 `Retry` / `Auth` / `Idempotency` / `Compensation`，减少外层消费方直接解析策略字符串 JSON 的需求
- 查询读侧当前也已开始把参数级策略补入结构化 `Policies.Parameter` 视图，显式返回 `allowedPaths` / `deniedPaths`
- 查询读侧当前也会把兼容保留的 flat 策略字段收敛到规范化 canonical 形态，避免把 legacy `retry-once` / `disabled` / `bearer` 一类旧值直接暴露给外层消费方
- 启动初始化当前也已开始复用同一套策略规范化规则，对旧存量 `RetryPolicy` / `AuthPolicy` / `IdempotencyPolicy` / `CompensationPolicy` / `ConsistencyMode` / `SideEffectLevel` 做自动收敛，尽量避免“新写入已规范、旧存量仍漂移”的双轨状态
- 但整体上 `ToolDefinition` 仍更接近“**HTTP webhook 工具定义 + 工具元数据**”，还不是完全通用的执行定义模型

### 4.2 ToolInvocation

目标字段：

- `invocation_id`
- `run_id`
- `step_id`
- `tool_name`
- `status`
- `input_payload`
- `output_payload`
- `error_payload`
- `idempotency_key`
- `callback_token`
- `executor_node`
- `started_at`
- `ended_at`

状态建议：

- `Pending`
- `Running`
- `Succeeded`
- `Failed`
- `Expired`
- `Cancelled`

当前实现状态：

- `tool_invocation` 表结构已存在于 SQL 设计中
- 当前主运行时已经开始将 `ToolInvocation` 接入异步工具回调主链路：
  - `tool-invocations/{invocationId}:complete` 会先查询 `ToolInvocation`
  - 校验 invocation 仍为 `Pending` 且 callback token 匹配
  - 再校验 invocation 关联 step 仍是当前 pending `tool_call`
- 现阶段异步工具挂起仍同时依赖：
  - `AgentRun.Status = WaitingTool`
  - `AgentStep` 中 `tool_call` 的 `Pending/Completed`
  - `waitToken` 兼容恢复入口

## 5. 副作用分级

- `read_only`
- `internal_write`
- `external_write`
- `destructive`

分级越高，越应要求审批、显式确认或人工接管。

> 当前实现状态：分级字段已落地，并已接入可配置审批策略 MVP；当前也已支持最小参数级审批策略，可按工具输入字段命中附加审批规则。

## 6. 调用模式

### 6.1 同步模式

- Runtime 调用后在当前执行片段拿到结果
- 适用于短耗时查询和确定性操作

当前实现状态：

- 已落地
- 同步工具执行完成后，`AgentRunLoop` 会写入 `tool_call` Completed step，并继续下一轮模型推理

### 6.2 异步模式

目标设计：

- 先创建 `ToolInvocation`
- Run 切换为 `WaitingTool`
- 等待回调或轮询结果

异步工具必须提供：

- `callback_token`
- 过期时间
- 幂等键

当前实现状态：

- 已支持异步挂起 / 恢复主链路，但实现方式更轻量：
  - 工具返回 `IsPending = true`
  - `AgentRun` 切换到 `WaitingTool`
  - 写入 Pending `tool_call` step
  - 通过 `POST /agent-runs/{runId}:resume` 恢复
- 当前已开始向 `ToolInvocation` 驱动模型演进，但 `resume` 兼容入口和 run 状态仍共同参与恢复

## 7. 标准结果结构

目标设计：所有工具结果归一化为：

```json
{
  "status": "succeeded",
  "data": {},
  "error": null,
  "meta": {
    "durationMs": 320,
    "source": "order-service",
    "cached": false
  }
}
```

当前实现状态：

- 当前 `HttpToolInvoker` 已开始兼容两类 webhook 返回协议：

```json
{
  "output": "...",
  "isPending": true,
  "waitToken": "..."
}
```

```json
{
  "status": "succeeded",
  "data": {},
  "error": null,
  "meta": {
    "waitToken": "..."
  }
}
```

- 当前已支持对 `succeeded / success / completed`、`pending / running / waiting`、`failed / error` 等标准状态做最小兼容解析
- Runtime 内部 `ToolExecutionResult` 当前已开始兼容承载 `status / error / meta`，同时保留 `Output / IsPending / WaitToken` 供现有运行时使用；主运行时、工具专用 complete 回调恢复以及审批放行后的工具执行当前也已能直接消费 `failed/error` 结果并统一落结构化失败步骤/调用审计；若工具定义声明了 `CompensationPolicy`，当前失败审计载荷会同步带出最小 `compensation.mode` 信息，且 `manual` 模式已可自动切入 `WaitingHuman`，但仍未完全演进为全链路统一标准结果对象
- 若返回体不匹配上述兼容协议，当前实现会尽量按原始文本返回

## 8. 幂等与补偿

目标设计：

- 每次调用都生成 `idempotency_key`
- 写操作工具至少满足“至少一次回调、至多一次生效”
- 不可逆副作用工具必须声明 `compensation_policy`
- 若工具侧无法保证幂等，Runtime 侧必须前置去重

当前实现状态：

- 相关字段已存在于 `ToolDefinition` / SQL 设计中
- 当前运行时已开始执行最小 HTTP webhook 重试策略
- 当前运行时已开始执行最小 webhook 幂等策略：可按 `IdempotencyPolicy` 生成并透传 `Idempotency-Key`
- 当前已落地最小 Runtime 前置去重：同 run 内启用 `IdempotencyPolicy` 的同工具、同输入，已完成结果可直接复用，pending 异步调用也会直接复用原等待态
- 当前补偿策略仍未真正进入自动执行链路，但失败步骤 / 调用审计当前已开始携带最小 `compensation.mode` 信息，为后续补偿执行编排预留稳定语义

## 9. 安全控制点

目标设计：

- 工具只允许从 allowlist 中解析
- 参数必须通过 schema 与黑白名单校验
- 敏感参数在日志中应脱敏
- destructive 工具默认要求审批

当前实现状态：

- `AgentRunLoop` 已基于 `AllowedTools` 做 allowlist 校验
- `ToolDefinition` 创建/更新阶段已做基础 JSON 合法性校验
- `ToolExecutor` 当前也已开始按 `ParameterPolicy` 对工具输入做路径级 allow/deny 校验
- 工具定义查询/创建/更新返回模型已开始对 `AuthHeaders`、webhook binding 内嵌认证头与 `CallbackSecret` 做最小脱敏
- 运行时查询/事件读侧与 `session_memory` 写回层当前已开始对常见敏感字段做最小递归 JSON 脱敏，覆盖 `GetAgentRunSteps`、`GetAgentRunApprovals`、`GetAgentRunEvents` 与 SSE `stream`
- 运行时输入 / 输出 schema 校验已落地最小常用子集与最小 `format` 校验；更完整的运行时日志全面脱敏以及更丰富的 destructive 工具治理策略仍未完整落地

## 10. 验收重点

目标验收：

- 同一工具调用重放不会重复生效
- 同步与异步模式都能回写标准结果
- 工具错误会被结构化记录并可被 Reflector 消费
- 高风险工具调用会触发正确的审批链路

当前实现验收重点：

- DB-first 工具解析语义正确
- webhook 工具优先于本地 handler 执行
- `inline_result` 工具可直接按 persisted binding 返回固定结果，不依赖 webhook 或本地 handler
- 未配置 endpoint 的内置工具仍可通过本地 handler 兼容执行
- 工具输入可按 `InputSchema` 的最小 JSON Schema 子集在执行前被拦截
- 工具同步完成结果可按 `OutputSchema` 的最小 JSON Schema 子集在执行后被拦截；异步 pending 结果仍不会在挂起当下即时校验，但在恢复继续 loop 或人工替代继续 loop 前会做最小后置输出校验
- 工具 schema 运行时校验当前已覆盖最小组合关键字与高级边界（`const`、`allOf`、`anyOf`、`oneOf`、`not`、`if/then/else`、`contains`、`minContains`、`maxContains`、`dependentRequired`、`dependentSchemas`、`patternProperties`、`additionalProperties` 对象形态、`exclusiveMinimum`、`exclusiveMaximum`）
- HTTP webhook 工具已开始兼容标准结果信封与 legacy `output/isPending/waitToken` 协议
- HTTP webhook 工具已开始按最小 `IdempotencyPolicy` 透传 `Idempotency-Key`
- `ToolDefinition` 写入阶段当前已开始规范化 `RetryPolicy`、`AuthPolicy`、`CompensationPolicy`、`ConsistencyMode` 与 `SideEffectLevel`，使 legacy 字符串输入与结构化策略在持久化层保持一致
- API 写侧当前也已开始支持结构化 `Policies` 请求体，并在 controller 层按逐字段合并方式归一到既有策略字符串字段；同一策略字段同时提供结构化值与旧 flat 字段时要求语义一致
- 启动初始化当前也会对旧存量策略字段做相同规则的归一化，保持命令写入与启动修复共用一套规范化语义
- 当前 `ConsistencyMode` 已收紧为有限枚举：`none`、`eventual`、`strong`
- 当前 `SideEffectLevel` 已收紧为有限枚举：`read_only`、`internal_write`、`external_write`、`destructive`
- Runtime 已开始对已完成幂等工具结果和 pending 异步调用做最小前置复用，避免同 run 内重复执行同工具、同输入调用
- HTTP webhook 工具可按最小 `RetryPolicy` 对超时、网络异常和可重试 HTTP 状态码进行重试
- 异步工具可进入 `WaitingTool`，并通过 `resume` 恢复完成 run
- 工具错误与输入校验错误当前已开始以最小结构化载荷回写到步骤流中：普通运行与审批放行后的同步工具失败会优先落成 `tool_call Failed`，并携带 `toolName` / `stage` / `message`
- 当前仍未达到完整统一的结构化错误模型
