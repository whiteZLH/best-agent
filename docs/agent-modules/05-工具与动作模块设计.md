# 工具与动作模块设计

## 1. 模块定位

工具与动作模块负责把 Agent 的结构化决策转成可执行动作。它承载工具定义、可见性解析、参数校验、权限控制、实际执行、超时重试、幂等与补偿。

> 当前实现状态：已落地统一工具执行入口 `IToolExecutor`，并通过 `AgentRunLoop -> ToolExecutor` 接入主运行时；但完整的工具解析、校验、策略编排仍未拆分成独立子模块。

## 2. 设计目标

- 工具注册与执行解耦
- 工具调用全程结构化、可审计
- 支持同步与异步模式
- 对副作用工具建立分级控制

> 当前实现状态：
> - “工具定义”和“工具执行”已经初步解耦：`ToolDefinition` 负责持久化定义，`ToolExecutor` / `HttpToolInvoker` 负责执行。
> - 当前更接近 **MVP 过渡形态**，不是最终平台态：数据库定义已成为主要事实来源，但本地工具与 HTTP 工具仍是并行模型。

## 3. 模块组成

### 3.1 Tool Registry

目标职责：管理系统内可用工具的执行绑定与本地实现。

当前实现状态：

- 当前 `ToolRegistry` **不负责管理所有 `ToolDefinition`**
- 它只负责管理进程内本地 handler 映射
- 当前内置了两个本地 handler：`echo_context`、`async_task`
- 在当前实现中，它主要承担兼容回退角色，而不是主工具目录

### 3.2 Tool Resolver

目标职责：根据 `AgentDefinition`、租户、环境、权限决定本次可见工具，并解析本次调用应绑定到哪个执行后端。

当前实现状态：

- 当前尚未形成独立 `ToolResolver` 组件
- 现阶段工具可用性校验主要体现在 `AgentRunLoop` 对 `AllowedTools` 的检查
- 工具执行解析当前固化在 `ToolExecutor` 中，语义为：
  1. 先读取 `ToolDefinition`
  2. 若定义禁用则拒绝执行
  3. 若定义配置了 `EndpointUrl`，走 HTTP webhook
  4. 若未配置 `EndpointUrl`，但本地存在 handler，则回退到本地 handler
  5. 若数据库无定义，但本地存在 handler，仍允许兼容执行

### 3.3 Tool Validator

目标职责：

- JSON Schema 校验
- 类型校验
- 安全校验

当前实现状态：

- 当前已实现 `ToolDefinition` 配置阶段的基础 JSON 校验：
  - `InputSchema`
  - `OutputSchema`
  - `AuthHeaders`
- 当前尚未形成运行时独立 `ToolValidator` 组件
- 当前执行前不会对 `toolInput` 做完整 schema 驱动校验

### 3.4 Tool Executor

目标职责：

- 实际执行工具
- 超时控制
- 重试控制
- 幂等控制

当前实现状态：

- 当前已实现统一执行入口 `ToolExecutor`
- `ToolExecutor` 采用 **DB-first** 语义：优先使用 `ToolDefinition` 决定执行方式
- 当前若命中 HTTP webhook 工具，则调用 `HttpToolInvoker`
- `HttpToolInvoker` 当前已覆盖：
  - 超时控制
  - AuthHeaders 透传
  - 输入 JSON 透传
  - webhook 响应解析
  - pending / waitToken 协议
- 当前尚未真正落地：
  - RetryPolicy 执行
  - IdempotencyPolicy 执行
  - CompensationPolicy 执行

### 3.5 Tool Policy

目标职责：负责副作用级别分级和审批联动。

当前实现状态：

- `ToolDefinition` 已持久化 `side_effect_level`
- 当前尚未形成独立 `ToolPolicy` 组件
- 当前尚未把副作用等级真正接到审批流或人工确认链路

## 4. 核心对象

### 4.1 ToolDefinition

目标字段：

- `tool_name`
- `description`
- `input_schema`
- `output_schema`
- `side_effect_level`
- `timeout_ms`
- `retry_policy`
- `auth_policy`
- `idempotency_policy`
- `async_supported`
- `consistency_mode`
- `compensation_policy`
- `enabled`

当前实现状态：

- 上述核心字段大多已经落地
- 当前实现额外包含 webhook 执行相关字段：
  - `endpoint_url`
  - `http_method`
  - `auth_headers`
- 因此当前 `ToolDefinition` 更接近“**HTTP webhook 工具定义 + 工具元数据**”，还不是完全通用的执行定义模型

### 4.2 ToolInvocation

目标字段：

- `invocation_id`
- `run_id`
- `step_id`
- `tool_name`
- `status`
- `input_payload`
- `output_payload`
- `error_payload`
- `idempotency_key`
- `callback_token`
- `executor_node`
- `started_at`
- `ended_at`

状态建议：

- `Pending`
- `Running`
- `Succeeded`
- `Failed`
- `Expired`
- `Cancelled`

当前实现状态：

- `tool_invocation` 表结构已存在于 SQL 设计中
- 但当前主运行时还**没有真正以 `ToolInvocation` 作为执行主状态对象**
- 现阶段异步工具挂起主要依赖：
  - `AgentRun.Status = WaitingTool`
  - `AgentStep` 中 `tool_call` 的 `Pending/Completed`
  - `waitToken + resume` 接口恢复

## 5. 副作用分级

- `read_only`
- `internal_write`
- `external_write`
- `destructive`

分级越高，越应要求审批、显式确认或人工接管。

> 当前实现状态：分级字段已落地，但审批联动尚未落地。

## 6. 调用模式

### 6.1 同步模式

- Runtime 调用后在当前执行片段拿到结果
- 适用于短耗时查询和确定性操作

当前实现状态：

- 已落地
- 同步工具执行完成后，`AgentRunLoop` 会写入 `tool_call` Completed step，并继续下一轮模型推理

### 6.2 异步模式

目标设计：

- 先创建 `ToolInvocation`
- Run 切换为 `WaitingTool`
- 等待回调或轮询结果

异步工具必须提供：

- `callback_token`
- 过期时间
- 幂等键

当前实现状态：

- 已支持异步挂起 / 恢复主链路，但实现方式更轻量：
  - 工具返回 `IsPending = true`
  - `AgentRun` 切换到 `WaitingTool`
  - 写入 Pending `tool_call` step
  - 通过 `POST /agent-runs/{runId}:resume` 恢复
- 当前还不是完整的 `ToolInvocation` 驱动模型

## 7. 标准结果结构

目标设计：所有工具结果归一化为：

```json
{
  "status": "succeeded",
  "data": {},
  "error": null,
  "meta": {
    "durationMs": 320,
    "source": "order-service",
    "cached": false
  }
}
```

当前实现状态：

- 当前尚未统一到上述标准结果结构
- 当前 `HttpToolInvoker` 主要识别的 webhook 返回协议是：

```json
{
  "output": "...",
  "isPending": true,
  "waitToken": "..."
}
```

- 若返回体不匹配标准对象结构，当前实现会尽量按原始文本返回

## 8. 幂等与补偿

目标设计：

- 每次调用都生成 `idempotency_key`
- 写操作工具至少满足“至少一次回调、至多一次生效”
- 不可逆副作用工具必须声明 `compensation_policy`
- 若工具侧无法保证幂等，Runtime 侧必须前置去重

当前实现状态：

- 相关字段已存在于 `ToolDefinition` / SQL 设计中
- 当前运行时尚未真正执行幂等控制、补偿策略和前置去重

## 9. 安全控制点

目标设计：

- 工具只允许从 allowlist 中解析
- 参数必须通过 schema 与黑白名单校验
- 敏感参数在日志中应脱敏
- destructive 工具默认要求审批

当前实现状态：

- `AgentRunLoop` 已基于 `AllowedTools` 做 allowlist 校验
- `ToolDefinition` 创建/更新阶段已做基础 JSON 合法性校验
- 运行时参数 schema 校验、敏感参数脱敏、destructive 工具审批联动尚未完整落地

## 10. 验收重点

目标验收：

- 同一工具调用重放不会重复生效
- 同步与异步模式都能回写标准结果
- 工具错误会被结构化记录并可被 Reflector 消费
- 高风险工具调用会触发正确的审批链路

当前实现验收重点：

- DB-first 工具解析语义正确
- webhook 工具优先于本地 handler 执行
- 未配置 endpoint 的内置工具仍可通过本地 handler 兼容执行
- 异步工具可进入 `WaitingTool`，并通过 `resume` 恢复完成 run
- 工具错误可回写到步骤流中，但尚未达到完整统一的结构化错误模型
