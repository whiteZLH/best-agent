# 记忆与检索模块设计

## 1. 模块定位

记忆与检索模块负责上下文补强，包括短期记忆、长期记忆、检索记忆和摘要记忆。它的核心目标是提升回答质量与执行连续性，但不能破坏当前用户输入的优先级。

> 当前实现状态：该模块已从纯设计态进入最小落地态。当前已补上 `knowledge_document`、`knowledge_chunk`、`embedding_index`、`session_memory`、`user_memory`、`summary_memory` 的持久化接入，并在运行时增加了最小上下文装配器，可在模型调用前注入 `summary_memory`、`session_memory`、`user_memory` 与 `knowledge_chunk`；当前这些读侧注入已开始由统一 `MemoryPolicy` 显式控制开关与数量，且该策略现已可通过 Agent Definition 创建 / 版本创建 API 写入。与此同时，Worker 已开始在可信完成点写入最小 memory：工具完成结果会按 `MemoryPolicy` 最小 allowlist 门禁写入 `session_memory`，run 完成后会写入模板化 `summary_memory`，工具结果中的显式结构化 `userMemories` 条目也可写入 `user_memory`；当前写侧还已拆分为独立开关，可分别控制 `session_memory`、`user_memory` 与 `summary_memory` 的写入。检索侧也已从“固定顺序取前 N 条”推进到最小 query-aware retrieval：支持轻量 query rewrite、候选召回、词法重排与 citation 注入；当前 `KnowledgeSources` 也已可通过 Agent Definition 创建 / 版本创建 API 写入并驱动运行时检索范围，但更复杂的记忆治理策略、更完整检索流水线和长期记忆分级写入仍未落地。

## 2. 设计目标

- 控制上下文长度与信息密度
- 让历史信息可追溯、可撤销、可过期
- 支持知识库检索与上下文注入
- 为多轮运行和恢复提供稳定上下文组装逻辑

> 当前实现状态：
> - 当前上下文组装已开始从轻量拼接演进，主要依赖：
>   - system prompt
>   - 当前输入
>   - 工具结果 follow-up input
>   - 可选 `summary_memory`
>   - 可选 `session_memory`
>   - 可选 `user_memory`
>   - 可选 `knowledge_chunk`
> - 当前上述可选上下文已开始由统一 `MemoryPolicy` 显式控制读取开关与条数上限，例如可关闭 `knowledge_chunk` 注入而保留 summary / memory 注入
> - 当前已形成“摘要记忆 + 检索片段”的最小装配骨架，但尚未形成短期记忆、长期记忆、检索、摘要的完整系统化链路。

## 3. 模块组成

### 3.1 短期记忆

目标适合存储：

- 最近 N 轮消息
- 最近一次工具结果
- 最近一段运行摘要

当前实现状态：

- 尚未落地独立短期记忆组件
- 当前最接近短期记忆的内容，是 Runtime 在下一轮 prompt 中拼接上一次工具结果
- 当前 `session_memory` 已可作为运行时上下文的一部分被注入，并受统一 `MemoryPolicy` 开关与条数限制控制
- 当前 Worker 已开始将可信工具完成结果按 `MemoryPolicy` 最小 allowlist 门禁写入 `session_memory`
- 当前仍没有独立消息缓冲 / 最近 N 轮消息重建机制

### 3.2 长期记忆

目标适合存储：

- 用户偏好
- 历史业务事实
- 结构化画像

当前实现状态：

- 存储层已落地 `user_memory`
- 当前运行时已可按 `tenant_id + user_id` 读取有限数量的 `user_memory` 注入上下文，并受统一 `MemoryPolicy` 开关与条数限制控制
- 当前已形成最小可信写入骨架：
  - 仅接受工具结果中显式声明的结构化 `userMemories`
  - 每条记录仍会记录 `source_type`、`source_ref`、`confidence`
  - 同一 `memory_key` 当前会按用户维度覆写更新
  - 读侧当前已统一按 `effective_at <= now < expires_at` 过滤，仅注入已生效且未过期的长期记忆
- 更完整的长期记忆治理、来源可信分级和审批联动仍未落地

### 3.3 检索记忆

目标适合存储：

- 文档库
- FAQ
- SOP
- 工单记录

当前实现状态：

- 存储层已落地 `knowledge_document` / `knowledge_chunk` / `embedding_index`
- 运行时已可按 `AgentDefinitionVersion.KnowledgeSources` 注入有限数量的 `knowledge_chunk`，并受统一 `MemoryPolicy` 开关与条数限制控制
- 当前已落地最小检索骨架：
  - Runtime 会基于当前输入构造 retrieval query
  - 对 `Original user input / Tool called / Tool result` 结构化 follow-up 输入可做轻量 query rewrite
  - Repository 会先召回候选，再按词法命中做最小 rerank
  - Runtime 当前也已开始为检索调用补最小 retrieval tracing/metrics
- `Hybrid Search` 和更强的语义 `Query Rewrite` / `Rerank` 仍未落地

### 3.4 摘要记忆

用于压缩长上下文，减少 token 消耗并保留历史线索。

当前实现状态：

- 存储层已落地 `summary_memory`
- 运行时已可读取当前 run / session 的最近有效 summary 注入模型上下文，并受统一 `MemoryPolicy` 开关控制
- 当前 run 完成时已可写入一条保守的模板化 summary
- 自动摘要生成与更丰富的更新策略仍未落地

## 4. 上下文装配顺序

目标建议固定顺序：

1. system/developer 指令
2. 当前用户输入
3. 最近 N 轮短期消息
4. 尚未过期的 summary
5. 必要的工具结果
6. 检索结果
7. 长期记忆

当前实现状态：

- 当前上下文装配顺序已开始靠近目标态，主要是：
  1. `system prompt`
  2. 当前输入
  3. 最近一次工具结果 follow-up input
  4. 尚未过期的 `summary_memory`
  5. 当前 session 的 `session_memory`
  6. 当前 user 的 `user_memory`
  7. `knowledge_chunk` 检索片段
- 当前仍没有消息级短期记忆重建和更丰富的 citation 注入

## 5. 长期记忆写入策略

目标建议至少满足：

- 只允许可信来源写入
- 每条记忆都记录 `source_type`、`source_ref`、`confidence`
- 支持 `ttl` 或失效策略
- 支持撤销或覆写
- 默认不写入模型的自由推测

建议将写入拆成独立决策：

- `should_write_memory`
- `memory_type`
- `memory_scope`
- `memory_value`
- `expires_at`

当前实现状态：

- 当前已落地最小可信写入骨架：
  - 工具完成结果可由 Worker 按 `MemoryPolicy` 最小 allowlist 门禁写入 `session_memory`
  - run 完成结果可由 Worker 写入模板化 `summary_memory`
- `user_memory` 当前也可由 Worker 写入，但仅限工具结果中显式声明的结构化 memory 条目
- `user_memory` 写入当前已同时支持绝对 `expiresAt` 和相对 `ttlSeconds`，并会在 `effectiveAt/expiresAt` 时间窗无效时丢弃该条候选
- 当前 `session_memory`、`user_memory`、`summary_memory` 的写入已支持分开控制，不再共享单一写侧开关
- handoff 子链路当前也已开始进入最小长期记忆边界治理：当 `RouteRule.context_scope` 或 handoff `context_overrides` 命中 `{"mode":"summary_only"}` 时，子 Run 除了关闭读侧上下文注入，也会同步关闭 `toolResultMemoryEnabled`、`userMemoryWriteEnabled` 与 `summaryMemoryWriteEnabled`
- handoff 子链路当前也已支持显式 `RouteRule.memory_scope={"mode":"read_only"}` 与 handoff `memory_overrides={"mode":"read_only"}`：会只关闭 `toolResultMemoryEnabled`、`userMemoryWriteEnabled` 与 `summaryMemoryWriteEnabled`，但保留 `summary/session/user/knowledge` 读侧上下文注入，便于“可读不可写”的子 Agent 协作
- 当前模型决策中仍没有 memory write 相关结构化动作
- `user_memory` 的更复杂覆写策略与审批联动仍未落地

## 6. 检索流水线

目标建议流程：

1. Query Rewrite
2. Hybrid Search
3. Rerank
4. Chunk Selection
5. Context Injection

当前实现状态：

- 已落地最小 `Chunk Selection + Context Injection` 骨架
- 已落地轻量 `Query Rewrite`
- `Hybrid Search` 与更强 `Rerank` 仍未落地

## 7. 检索结果结构

目标建议字段：

- `document_id`
- `chunk_id`
- `content`
- `score`
- `source`
- `metadata`

当前实现状态：

- 已有 `knowledge_chunk` 实体承载 `document_id`、`content`、`source`、`metadata`
- 当前已在最小链路中形成隐式 lexical score，但尚未抽象成独立 score / rerank 结果对象

## 8. 注入策略

目标建议：

- 直接拼接上下文
- 结构化 citation
- 摘要后拼接

注入时必须携带来源元数据，便于回答阶段引用和审计。

当前实现状态：

- 已支持直接拼接 summary 与 knowledge chunk 到模型输入
- 当前 prompt 注入已附带最小 citation / source 元数据
- 当前已支持最小回答阶段 `References` 追加；更完整的结构化 citation 协议与检索审计注入仍未落地

## 9. 风险控制点

目标设计：

- 长期记忆不得覆盖用户当前显式输入
- summary 需要记录生成时点，避免递归摘要污染
- 检索结果必须与租户和权限隔离一致
- 记忆写入应与审批策略协同，避免错误事实固化

当前实现状态：

- 检索结果当前已开始进入运行时主链路，因此租户隔离和摘要污染风险已进入实现面
- 当前最小实现主要通过 `tenant_id` 和过期时间过滤降低风险，但还没有完整的权限隔离、写入审批联动和摘要更新策略

## 10. 数据存储建议

目标对象：

- `session_memory`
- `user_memory`
- `summary_memory`
- `knowledge_document`
- `knowledge_chunk`
- `embedding_index`

当前实现状态：

- 已落地
- 当前运行时已依赖 `summary_memory`、`session_memory`、`user_memory` 与 `knowledge_chunk`
- `embedding_index` 目前仍主要提供后续向量检索扩展的持久化基础

## 11. 验收重点

目标验收：

- 上下文超过阈值时可自动摘要而非直接截断
- 长期记忆可按来源、置信度和 TTL 管理
- 检索结果可标准化注入 Planner 和回答阶段
- 记忆与检索均符合租户隔离和审计要求

当前实现状态：

- 当前已进入“最小可验收落地状态”：
- `summary_memory` 与 `knowledge_chunk` 已可注入运行时上下文
- 相关持久化对象、仓储和依赖注入已接通
- 后续应继续扩展为真正的 summary / retrieval / memory write / citation 链路
