# 记忆与检索模块设计

## 1. 模块定位

记忆与检索模块负责上下文补强，包括短期记忆、长期记忆、检索记忆和摘要记忆。它的核心目标是提升回答质量与执行连续性，但不能破坏当前用户输入的优先级。

> 当前实现状态：该模块目前基本仍处于目标设计阶段。当前运行时尚未落地独立的记忆系统、知识库检索流水线或上下文压缩摘要机制。

## 2. 设计目标

- 控制上下文长度与信息密度
- 让历史信息可追溯、可撤销、可过期
- 支持知识库检索与上下文注入
- 为多轮运行和恢复提供稳定上下文组装逻辑

> 当前实现状态：
> - 当前上下文组装仍然非常轻量，主要依赖：
>   - system prompt
>   - 当前输入
>   - 工具结果 follow-up input
> - 尚未形成短期记忆、长期记忆、检索、摘要的系统化装配链路。

## 3. 模块组成

### 3.1 短期记忆

目标适合存储：

- 最近 N 轮消息
- 最近一次工具结果
- 最近一段运行摘要

当前实现状态：

- 尚未落地独立短期记忆组件
- 当前最接近短期记忆的内容，是 Runtime 在下一轮 prompt 中拼接上一次工具结果
- 当前没有独立消息缓冲 / 最近 N 轮消息重建机制

### 3.2 长期记忆

目标适合存储：

- 用户偏好
- 历史业务事实
- 结构化画像

当前实现状态：

- 尚未落地

### 3.3 检索记忆

目标适合存储：

- 文档库
- FAQ
- SOP
- 工单记录

当前实现状态：

- 尚未落地

### 3.4 摘要记忆

用于压缩长上下文，减少 token 消耗并保留历史线索。

当前实现状态：

- 尚未落地

## 4. 上下文装配顺序

目标建议固定顺序：

1. system/developer 指令
2. 当前用户输入
3. 最近 N 轮短期消息
4. 尚未过期的 summary
5. 必要的工具结果
6. 检索结果
7. 长期记忆

当前实现状态：

- 当前上下文装配顺序远比目标态简单，主要是：
  1. `system prompt`
  2. 当前输入
  3. 工具结果 follow-up input
- 当前没有短期消息装配、summary 注入、检索结果注入、长期记忆注入

## 5. 长期记忆写入策略

目标建议至少满足：

- 只允许可信来源写入
- 每条记忆都记录 `source_type`、`source_ref`、`confidence`
- 支持 `ttl` 或失效策略
- 支持撤销或覆写
- 默认不写入模型的自由推测

建议将写入拆成独立决策：

- `should_write_memory`
- `memory_type`
- `memory_scope`
- `memory_value`
- `expires_at`

当前实现状态：

- 尚未落地
- 当前模型决策中也没有 memory write 相关结构化动作

## 8. 注入策略

目标建议：

- 直接拼接上下文
- 结构化 citation
- 摘要后拼接

注入时必须携带来源元数据，便于回答阶段引用和审计。

当前实现状态：

- 尚未落地
- 当前没有 citation / 检索元数据注入机制

## 9. 风险控制点

目标设计：

- 长期记忆不得覆盖用户当前显式输入
- summary 需要记录生成时点，避免递归摘要污染
- 检索结果必须与租户和权限隔离一致
- 记忆写入应与审批策略协同，避免错误事实固化

当前实现状态：

- 这些风险控制点目前基本仍停留在设计层
- 当前由于尚未落地记忆与检索系统，也尚未进入对应主链路风险面

## 11. 验收重点

目标验收：

- 上下文超过阈值时可自动摘要而非直接截断
- 长期记忆可按来源、置信度和 TTL 管理
- 检索结果可标准化注入 Planner 和回答阶段
- 记忆与检索均符合租户隔离和审计要求

当前实现状态：

- 当前该模块尚未进入可验收落地状态
- 后续真正实现时，可优先复用当前 Runtime 已验证过的上下文拼接路径，再逐步扩展为 summary / retrieval / memory injection 链路
