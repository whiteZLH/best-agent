# 记忆与检索模块设计

## 1. 模块定位

记忆与检索模块负责上下文补强，包括短期记忆、长期记忆、检索记忆和摘要记忆。它的核心目标是提升回答质量与执行连续性，但不能破坏当前用户输入的优先级。

## 2. 设计目标

- 控制上下文长度与信息密度
- 让历史信息可追溯、可撤销、可过期
- 支持知识库检索与上下文注入
- 为多轮运行和恢复提供稳定上下文组装逻辑

## 3. 模块组成

### 3.1 短期记忆

适合存储：

- 最近 N 轮消息
- 最近一次工具结果
- 最近一段运行摘要

### 3.2 长期记忆

适合存储：

- 用户偏好
- 历史业务事实
- 结构化画像

### 3.3 检索记忆

适合存储：

- 文档库
- FAQ
- SOP
- 工单记录

### 3.4 摘要记忆

用于压缩长上下文，减少 token 消耗并保留历史线索。

## 4. 上下文装配顺序

建议固定顺序：

1. system/developer 指令
2. 当前用户输入
3. 最近 N 轮短期消息
4. 尚未过期的 summary
5. 必要的工具结果
6. 检索结果
7. 长期记忆

## 5. 长期记忆写入策略

建议至少满足：

- 只允许可信来源写入
- 每条记忆都记录 `source_type`、`source_ref`、`confidence`
- 支持 `ttl` 或失效策略
- 支持撤销或覆写
- 默认不写入模型的自由推测

建议将写入拆成独立决策：

- `should_write_memory`
- `memory_type`
- `memory_scope`
- `memory_value`
- `expires_at`

## 6. 检索流水线

建议流程：

1. Query Rewrite
2. Hybrid Search
3. Rerank
4. Chunk Selection
5. Context Injection

## 7. 检索结果结构

建议字段：

- `document_id`
- `chunk_id`
- `content`
- `score`
- `source`
- `metadata`

## 8. 注入策略

- 直接拼接上下文
- 结构化 citation
- 摘要后拼接

注入时必须携带来源元数据，便于回答阶段引用和审计。

## 9. 风险控制点

- 长期记忆不得覆盖用户当前显式输入
- summary 需要记录生成时点，避免递归摘要污染
- 检索结果必须与租户和权限隔离一致
- 记忆写入应与审批策略协同，避免错误事实固化

## 10. 数据存储建议

- `session_memory`
- `user_memory`
- `summary_memory`
- `knowledge_document`
- `knowledge_chunk`
- `embedding_index`

## 11. 验收重点

- 上下文超过阈值时可自动摘要而非直接截断
- 长期记忆可按来源、置信度和 TTL 管理
- 检索结果可标准化注入 Planner 和回答阶段
- 记忆与检索均符合租户隔离和审计要求
