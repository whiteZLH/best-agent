# 模型网关模块设计

## 1. 模块定位

模型网关模块为平台提供统一的 `ModelGateway`，屏蔽不同模型供应商的接入差异，并向上游输出统一的文本、结构化 JSON、tool calls、reasoning summary 和 finish reason。

> 当前实现状态：当前已落地一个统一的 `IModelGateway` 抽象和 `OpenAiCompatibleModelGateway` 实现，但能力仍偏 MVP，只稳定支持文本输入 / 文本输出，不是完整的平台化模型网关。

## 2. 设计目标

- 标准化模型接入协议
- 降低 Planner 与具体模型 SDK 的耦合
- 统一采集 token、成本和延迟
- 为多模型切换和灰度提供基础

> 当前实现状态：
> - 当前已实现对具体供应商 HTTP 协议的封装，运行时不直接依赖模型 SDK。
> - 当前支持通过 Definition 传入模型名，具备最基本的“按 Agent 切模型”能力。
> - 当前也已开始最小采集 usage / 成本：OpenAI 兼容响应中的 `usage` 会进入统一结果对象，成本可按配置的 token 单价换算；同时 `model_call` step 已开始记录最小 `model/tokens/cost` 审计信息，延迟则继续复用 step 的 `started_at/ended_at/duration_ms`。
> - 多供应商切换和更完整灰度能力仍未落地。

## 3. 核心职责

- 供应商适配
- 请求格式标准化
- 响应格式标准化
- Token 和成本统计
- 失败归类与错误归一化

当前实现状态：

- 当前已落地：
  - 供应商适配（OpenAI 兼容接口）
  - 基础请求格式标准化
  - 基础错误抛出与空响应校验
  - 最小 `message.content` 字符串 / 文本数组片段兼容解析
- 当前也已开始最小落地：
  - `usage` -> 统一结果对象的 token 统计
  - 基于配置单价的最小成本换算
  - 通过 `AgentStep` 的时间字段承载最小调用延迟审计
- 当前尚未完整落地：
  - 更完整的统一响应结构标准化
  - 独立模型调用审计实体
  - 失败分类枚举化

## 4. 标准输入

目标建议至少包含：

- 消息列表
- 模型标识
- 输出模式
- 输出 schema
- 工具定义
- 超时配置
- 温度等生成参数

当前实现状态：

- 当前 `GenerateTextRequest` 输入模型较轻，主要包含：
  - `model`
  - `system prompt`
  - `input`
- 当前已开始补上最小多消息、超时、输出格式、工具定义与生成参数能力：`GenerateTextRequest.Messages` 显式传入时会优先按多消息请求下发，当前消息角色已开始收紧到 OpenAI 兼容 `chat/completions` 的 `developer/system/user/assistant/tool` 集合；单条 `GenerateTextMessage` 除 `role/content` 外也已开始支持最小 `name` 与 `tool_call_id`，其中 `tool` 角色消息要求显式携带 `tool_call_id`，同时 `content` 当前也已开始支持字符串与最小结构化 content parts（`text` / `image_url` / `input_audio` / `file`），`assistant` 角色消息当前也已开始支持显式 `tool_calls` 历史载荷；显式传入的消息列表若全部无效，也会直接拒绝而不再静默回退到 `system prompt + input`；可通过 `GenerateTextRequest.OutputMode` / `GenerateTextRequest.OutputSchema` 请求结构化输出，其中显式 `OutputMode=text` 当前也已开始下发 `response_format={type:\"text\"}`，版本级 `OutputSchema` 也已开始透传到模型网关，`json_schema` 形态当前也已支持通过 `GenerateTextRequest.OutputName` / `GenerateTextRequest.OutputDescription` / `GenerateTextRequest.OutputStrict` 定制 schema 名称、描述与 strict 开关；`GenerateTextRequest.MaxOutputTokens` 当前也已对齐映射到 OpenAI 兼容 `chat/completions` 的 `max_completion_tokens`；`GenerateTextRequest.Tools` 当前也已开始支持最小工具定义下发，`AgentRunLoop` 会按版本级 `AllowedTools` 解析 enabled `ToolDefinition` 并向 OpenAI 兼容请求下发最小 `tools.function.parameters`，并显式补上 `function.strict`；请求侧若显式传入 `Tools`，当前要求至少存在一个具名工具定义；同时存在工具时默认下发 `tool_choice = auto`，并额外默认下发 `parallel_tool_calls = false` 以贴合当前单工具调用 Runtime 约束，也可由 `GenerateTextRequest.ToolChoice` 显式指定 `auto/none/required/具体工具名`，其中 `auto|required|具体工具名` 当前都要求至少存在有效声明工具；同时 `GenerateTextRequest.UserId` 当前也已开始支持把运行时 `user_id` 透传给 OpenAI 兼容请求的 `user` 字段，便于最小下游审计与归因；`GenerateTextRequest.Metadata` 当前也已开始支持最小请求级元数据下发，`AgentRunLoop` 会自动附带 `run_id/agent_code/version_id/tenant_id/user_id/session_id/root_run_id/parent_run_id` 等最小运行上下文，便于下游查询和审计；`GenerateTextRequest.ServiceTier` 与 `GenerateTextRequest.Store` 当前也已开始支持把请求级 `service_tier` 与 `store` 透传给 OpenAI 兼容请求；此外也可通过 `OpenAI:TimeoutSeconds`、`OpenAI:Temperature`、`OpenAI:MaxOutputTokens`、`OpenAI:TopP`、`OpenAI:PresencePenalty`、`OpenAI:FrequencyPenalty`、`OpenAI:LogitBias`、`OpenAI:Seed`、`OpenAI:StopSequences`、`OpenAI:ParallelToolCalls`、`OpenAI:ReasoningEffort`、`OpenAI:Verbosity`、`OpenAI:ServiceTier` 与 `OpenAI:Store` 提供全局默认值，也可由 `GenerateTextRequest.TimeoutSeconds` / `GenerateTextRequest.Temperature` / `GenerateTextRequest.MaxOutputTokens` / `GenerateTextRequest.TopP` / `GenerateTextRequest.PresencePenalty` / `GenerateTextRequest.FrequencyPenalty` / `GenerateTextRequest.LogitBias` / `GenerateTextRequest.Seed` / `GenerateTextRequest.StopSequences` / `GenerateTextRequest.ParallelToolCalls` / `GenerateTextRequest.ReasoningEffort` / `GenerateTextRequest.Verbosity` / `GenerateTextRequest.ServiceTier` / `GenerateTextRequest.Store` 做请求级覆盖
- 当前请求最终映射为 OpenAI 兼容 `chat/completions`
- 当前尚未直接支持：
  - 更丰富的工具定义下发与 tool choice 策略
  - 更丰富的输出 schema / response format 策略
  - 更丰富的输出模式切换
  - 除温度、输出 token 上限、`top_p` 与 penalty 参数外的更多细粒度生成参数配置

## 5. 标准输出

目标输出：

- 文本回答
- 结构化 JSON
- `tool calls`
- `reasoning summary`
- `finish reason`

补充约束：

- `tool calls` 必须带结构化参数
- `reasoning summary` 仅保留可审计摘要
- `finish reason` 建议枚举化，如 `completed`、`tool_call`、`handoff`、`max_turns`

当前实现状态：

- 当前 `GenerateTextResult` 已开始稳定承载：
  - 文本输出
  - `prompt/completion/total tokens`
  - 最小 `cost`
- 当前也已开始把 OpenAI 兼容响应中的 `finish_reason` 规范化为平台侧最小枚举（如 `completed` / `tool_call` / `max_output_tokens` / `content_filtered`），并把最小 `reasoning_summary/reasoning`、响应 `id`、实际 `service_tier` 与原生 `tool_calls` 列表显式归一到 `GenerateTextResult` 与 `model_call` 审计 payload
- 当前 tool call 主链路仍主要通过模型返回 JSON 文本，再由上层 `IStepDecisionParser` 解析为结构化决策；同时 OpenAI 兼容网关也已开始支持把原生单个 `tool_calls` 响应归一回现有 `{"action":"tool_call",...}` JSON 决策输出
- 当前尚未完整落地：
  - 多个原生 tool call 的并行执行

## 6. 与 Planner 的关系

- Planner 负责决策语义
- ModelGateway 负责供应商适配
- Planner 不直接依赖具体模型 SDK
- Runtime 可以基于策略选择不同模型再交给网关执行

当前实现状态：

- 当前尚未落地独立 Planner 模块
- 当前决策语义主要体现在：
  - `AgentRunLoop` 构造 prompt
  - 模型返回 JSON 文本
  - `IStepDecisionParser` 解释该 JSON
- `IModelGateway` 仍保持了和具体 SDK/HTTP 适配的边界分离

## 7. 监控与审计

目标建议记录：

- 请求模型
- 输入输出 token
- 延迟
- 成本
- 失败类型
- 响应模式

对应日志建议写入：

- `model_call_log`

当前实现状态：

- 当前尚未落地独立 `model_call_log`
- 当前对模型调用的运行痕迹主要体现在：
  - `AgentStep` 中的 `model_call`
  - `model_call` step 的最小 typed usage/cost 审计视图
  - 错误时抛出的异常信息
- token / 成本 / 延迟维度当前已具备最小审计闭环：
  - token / 成本：进入 `GenerateTextResult`，并写入 `model_call` step 的 `DecisionPayload`
  - 成本：累计到 `AgentRun.TotalCost`
  - 延迟：通过 `started_at` / `ended_at` / `duration_ms` 体现
- 但按模型维度聚合统计、独立日志实体与更细粒度观测仍未落地

## 8. 风险控制点

- 不接受无法结构化校验的工具调用输出
- 不依赖完整链路思维文本作为恢复依据
- 不让上层逻辑感知供应商专有细节
- 供应商故障时要可降级到备用模型或失败出口

当前实现状态：

- 当前已部分满足：
  - 上层逻辑不直接依赖供应商细节
  - Runtime 只消费文本输出，并通过 `IStepDecisionParser` 做结构化解释
- 当前也已开始补上最小原生工具调用结构校验：OpenAI 兼容响应中的原生 `tool_calls` 若命中当前声明工具列表，且 `arguments` 为可解析 JSON 对象，才会继续归一进入现有 Runtime
- 当前尚未落地：
  - 备用模型降级
  - 完整失败出口策略

## 9. 验收重点

目标验收：

- 多供应商模型能输出统一结构
- Planner 能基于统一网关切换模型
- 成本和延迟可按 Run、Step 和模型维度统计
- 工具调用、结构化 JSON、普通回答三类输出都可稳定支持

当前实现验收重点：

- OpenAI 兼容模型可通过统一网关稳定返回文本结果
- Runtime 可根据 Definition 中的 `default_model` 选择模型
- 文本回答与 JSON 决策输出可稳定支持
- 多供应商统一结构、原生 tool calls、成本与延迟审计仍属于后续目标态
