# 模型网关模块设计

## 1. 模块定位

模型网关模块为平台提供统一的 `ModelGateway`，屏蔽不同模型供应商的接入差异，并向上游输出统一的文本、结构化 JSON、tool calls、reasoning summary 和 finish reason。

## 2. 设计目标

- 标准化模型接入协议
- 降低 Planner 与具体模型 SDK 的耦合
- 统一采集 token、成本和延迟
- 为多模型切换和灰度提供基础

## 3. 核心职责

- 供应商适配
- 请求格式标准化
- 响应格式标准化
- Token 和成本统计
- 失败归类与错误归一化

## 4. 标准输入

建议至少包含：

- 消息列表
- 模型标识
- 输出模式
- 输出 schema
- 工具定义
- 超时配置
- 温度等生成参数

## 5. 标准输出

- 文本回答
- 结构化 JSON
- `tool calls`
- `reasoning summary`
- `finish reason`

补充约束：

- `tool calls` 必须带结构化参数
- `reasoning summary` 仅保留可审计摘要
- `finish reason` 建议枚举化，如 `completed`、`tool_call`、`handoff`、`max_turns`

## 6. 与 Planner 的关系

- Planner 负责决策语义
- ModelGateway 负责供应商适配
- Planner 不直接依赖具体模型 SDK
- Runtime 可以基于策略选择不同模型再交给网关执行

## 7. 监控与审计

建议记录：

- 请求模型
- 输入输出 token
- 延迟
- 成本
- 失败类型
- 响应模式

对应日志建议写入：

- `model_call_log`

## 8. 风险控制点

- 不接受无法结构化校验的工具调用输出
- 不依赖完整链路思维文本作为恢复依据
- 不让上层逻辑感知供应商专有细节
- 供应商故障时要可降级到备用模型或失败出口

## 9. 验收重点

- 多供应商模型能输出统一结构
- Planner 能基于统一网关切换模型
- 成本和延迟可按 Run、Step 和模型维度统计
- 工具调用、结构化 JSON、普通回答三类输出都可稳定支持
