# 模型网关模块设计

## 1. 模块定位

模型网关模块为平台提供统一的 `ModelGateway`，屏蔽不同模型供应商的接入差异，并向上游输出统一的文本、结构化 JSON、tool calls、reasoning summary 和 finish reason。

> 当前实现状态：当前已落地一个统一的 `IModelGateway` 抽象和 `OpenAiCompatibleModelGateway` 实现，但能力仍偏 MVP，只稳定支持文本输入 / 文本输出，不是完整的平台化模型网关。

## 2. 设计目标

- 标准化模型接入协议
- 降低 Planner 与具体模型 SDK 的耦合
- 统一采集 token、成本和延迟
- 为多模型切换和灰度提供基础

> 当前实现状态：
> - 当前已实现对具体供应商 HTTP 协议的封装，运行时不直接依赖模型 SDK。
> - 当前支持通过 Definition 传入模型名，具备最基本的“按 Agent 切模型”能力。
> - 当前尚未统一采集 token、成本、延迟，也未形成多供应商切换和灰度能力。

## 3. 核心职责

- 供应商适配
- 请求格式标准化
- 响应格式标准化
- Token 和成本统计
- 失败归类与错误归一化

当前实现状态：

- 当前已落地：
  - 供应商适配（OpenAI 兼容接口）
  - 基础请求格式标准化
  - 基础错误抛出与空响应校验
- 当前尚未完整落地：
  - 统一响应结构标准化
  - token / 成本统计
  - 延迟统计
  - 失败分类枚举化

## 4. 标准输入

目标建议至少包含：

- 消息列表
- 模型标识
- 输出模式
- 输出 schema
- 工具定义
- 超时配置
- 温度等生成参数

当前实现状态：

- 当前 `GenerateTextRequest` 输入模型较轻，主要包含：
  - `model`
  - `system prompt`
  - `input`
- 当前请求最终映射为 OpenAI 兼容 `chat/completions`
- 当前尚未直接支持：
  - 工具定义下发
  - 输出 schema
  - 输出模式切换
  - 温度等更细粒度生成参数配置

## 5. 标准输出

目标输出：

- 文本回答
- 结构化 JSON
- `tool calls`
- `reasoning summary`
- `finish reason`

补充约束：

- `tool calls` 必须带结构化参数
- `reasoning summary` 仅保留可审计摘要
- `finish reason` 建议枚举化，如 `completed`、`tool_call`、`handoff`、`max_turns`

当前实现状态：

- 当前 `GenerateTextResult` 只稳定承载文本输出
- 当前 tool call 不是由网关以原生结构输出，而是通过模型返回 JSON 文本，再由上层 `IStepDecisionParser` 解析为结构化决策
- 当前尚未显式返回：
  - reasoning summary
  - finish reason
  - 原生 tool call 列表

## 6. 与 Planner 的关系

- Planner 负责决策语义
- ModelGateway 负责供应商适配
- Planner 不直接依赖具体模型 SDK
- Runtime 可以基于策略选择不同模型再交给网关执行

当前实现状态：

- 当前尚未落地独立 Planner 模块
- 当前决策语义主要体现在：
  - `AgentRunLoop` 构造 prompt
  - 模型返回 JSON 文本
  - `IStepDecisionParser` 解释该 JSON
- `IModelGateway` 仍保持了和具体 SDK/HTTP 适配的边界分离

## 7. 监控与审计

目标建议记录：

- 请求模型
- 输入输出 token
- 延迟
- 成本
- 失败类型
- 响应模式

对应日志建议写入：

- `model_call_log`

当前实现状态：

- 当前尚未落地独立 `model_call_log`
- 当前对模型调用的运行痕迹主要体现在：
  - `AgentStep` 中的 `model_call`
  - 错误时抛出的异常信息
- token / 成本 / 延迟维度的细粒度审计仍未落地

## 8. 风险控制点

- 不接受无法结构化校验的工具调用输出
- 不依赖完整链路思维文本作为恢复依据
- 不让上层逻辑感知供应商专有细节
- 供应商故障时要可降级到备用模型或失败出口

当前实现状态：

- 当前已部分满足：
  - 上层逻辑不直接依赖供应商细节
  - Runtime 只消费文本输出，并通过 `IStepDecisionParser` 做结构化解释
- 当前尚未落地：
  - 备用模型降级
  - 原生工具调用结构校验
  - 完整失败出口策略

## 9. 验收重点

目标验收：

- 多供应商模型能输出统一结构
- Planner 能基于统一网关切换模型
- 成本和延迟可按 Run、Step 和模型维度统计
- 工具调用、结构化 JSON、普通回答三类输出都可稳定支持

当前实现验收重点：

- OpenAI 兼容模型可通过统一网关稳定返回文本结果
- Runtime 可根据 Definition 中的 `default_model` 选择模型
- 文本回答与 JSON 决策输出可稳定支持
- 多供应商统一结构、原生 tool calls、成本与延迟审计仍属于后续目标态
