# 规划与推理模块设计

## 1. 模块定位

规划与推理模块负责把上下文转成下一步可执行动作。它不是自由聊天层，而是 Runtime 的结构化决策引擎，输出必须符合 `PlanDecision` 契约。

> 当前实现状态：当前尚未落地独立 Planner / Reflector 模块。现阶段更接近一个轻量的“结构化决策解析层”：模型输出 JSON 文本后，由 `IStepDecisionParser` 解析为 `StepDecision`，再交由 Runtime 执行。

## 2. 设计目标

- 输出结构化而不是自由文本决策
- 将“思考”和“执行”职责解耦
- 允许模型建议，但最终由 Runtime 裁剪和确认
- 为简单场景和复杂场景提供同构扩展路径

> 当前实现状态：
> - “输出结构化而不是自由文本”已部分落地：当前要求模型输出 `respond` 或 `tool_call` JSON。
> - “思考”和“执行”已初步解耦：模型只给决策，Runtime 决定是否执行工具。
> - 当前尚未形成复杂场景下的 Planner / Reflector / 多阶段推理架构。

## 3. 模块组成

### 3.1 Planner

目标职责：

- 判断是否需要工具
- 判断是否需要检索
- 判断是否需要 handoff
- 判断是否可直接回答

当前实现状态：

- 当前尚未落地独立 `Planner`
- 当前模型决策能力主要体现在 Runtime 提示词中，由模型直接返回：
  - `respond`
  - `tool_call`
  - `handoff`
- 当前 `handoff` 已支持最小路由元数据：
  - `target_agent`
  - `confidence`
  - `reason`
  - `context_overrides`
  - `tool_overrides`
  - `approval_required`
- 当前已开始支持最小 `retrieve` / `request_human` / `fail` 决策类型，但 `request_approval` 与更完整的 Planner 决策协议仍未落地

### 3.2 Executor

目标职责：

- 执行工具、检索、handoff 对应动作
- 收集并归一化结果
- 统一错误处理

当前实现状态：

- 当前已存在“执行”能力，但未以规划模块子组件名义独立实现
- 当前执行主要由 Runtime + `IToolExecutor` 共同完成：
  - Runtime 负责读取决策、校验 allowlist、推进 step
  - ToolExecutor 负责真正执行工具
- 当前已开始支持最小 retrieval / handoff 执行路径：`retrieve` 会生成 retrieval follow-up 并继续下一轮模型调用，`handoff` 会按模式进入父子 Run 协同；但二者仍未抽成独立 Executor 子组件

### 3.3 Reflector

目标职责：

- 判断当前结果是否足够
- 判断是否需要重试或降级
- 判断是否结束或转人工

当前实现状态：

- 当前尚未落地独立 `Reflector`
- 当前“是否结束”主要依赖模型下一轮返回 `respond`
- 当前没有显式的重试 / 降级 / 转人工反思层

## 4. PlanDecision 设计

目标建议字段：

- `decision_id`
- `type`
- `reason`
- `confidence`
- `selected_model`
- `response_message`
- `tool_calls`
- `retrieval_request`
- `handoff_request`
- `approval_request`
- `terminate_reason`

`type` 枚举建议：

- `respond`
- `tool_call`
- `retrieve`
- `handoff`
- `request_approval`
- `request_human`
- `fail`

当前实现状态：

- 当前尚未采用完整 `PlanDecision` 对象
- 当前实际落地的是轻量 `StepDecision`：
  - `Action`
  - `Response`
  - `ToolName`
  - `ToolInput`
  - `RetrievalQuery`
  - `TargetAgent`
  - `HandoffInput`
  - `HandoffMode`
  - `HandoffReason`
  - `HandoffConfidence`
  - `HandoffContextOverrides`
  - `HandoffToolOverrides`
  - `HandoffApprovalRequired`
- 当前支持的动作包括：
  - `respond`
  - `retrieve`
  - `tool_call`
  - `handoff`
  - `request_human`
  - `fail`

## 5. 关键约束

目标约束：

- 任一时刻只允许一个主 `type`
- 多工具调用必须声明是否允许并行
- handoff 必须包含目标 Agent、上下文范围和等待模式
- `fail` 必须返回结构化错误码
- Planner 不得绕过审批或能力边界

当前实现状态：

- 当前已自然满足“任一时刻一个主动作”，因为 `StepDecision` 只能表示一个动作
- 当前不支持多工具并行计划
- 当前已支持最小 `handoff` / `request_human` / `fail` 结构化决策类型
- 当前 Runtime 会对工具能力边界做 allowlist 校验，不允许模型直接越权调用工具

## 6. 执行优先级

目标建议固定如下优先级：

1. 租户级安全策略
2. AgentDefinition 能力边界
3. Runtime 当前状态约束
4. 审批策略
5. Planner 输出建议

如果低优先级与高优先级冲突，由 Runtime 拒绝并生成降级或失败决策。

当前实现状态：

- 当前最明确落地的是：
  1. AgentDefinition 的 `AllowedTools`
  2. Runtime 的 `WaitingTool` / `Running` 状态约束
  3. `waitToken` 恢复校验
  4. 模型输出建议
- 租户级安全策略、审批策略、结构化降级决策尚未形成完整优先级系统

## 7. 输入输出接口

目标输入：

- 当前 Run 状态
- 上下文消息
- AgentDefinition
- 可见工具集
- 检索结果
- 已有 Step 与 Tool Result

目标输出：

- 标准 `PlanDecision`
- 可审计的简短 reason
- 可选的 `selected_model`

当前实现状态：

- 当前输入主要是：
  - 当前 Run 输入
  - 当前 AgentDefinitionVersion
  - 上一轮 tool result 拼接出的 follow-up input
- 当前还没有独立消息上下文重建、检索结果输入、显式 selected model 输出
- 当前输出是 `StepDecision`

## 8. 落地实现建议

目标建议：

- Planner 初期可以与模型调用一一对应
- Reflector 可在 Phase 1 以规则优先实现
- 复杂场景再拆分为独立策略模块
- 所有模型输出都先经 JSON Schema 校验

当前实现状态：

- 当前确实采取了“模型调用一一对应轻量决策”的 MVP 路径
- 当前还没有独立 Reflector
- 当前模型输出先由 `StepDecisionParser` 做 JSON 解析，但还没有严格 JSON Schema 校验
- 如果 JSON 不可解析，当前 parser 会退化为把整段文本当成 `respond`

## 9. 风险控制点

目标设计：

- 不接受“模糊工具意图”作为最终输出
- 不依赖完整链路思维作为恢复依据
- 模型失败时要有 `fail` 或 `request_human` 降级出口
- 并行工具计划需要由 Runtime 再次裁剪

当前实现状态：

- 当前已部分满足：
  - 不依赖完整链路思维作为恢复依据
  - Runtime 会再次校验工具 allowlist
- 当前已支持最小：
  - `fail` / `request_human` 结构化降级决策
  - 并行工具计划裁剪
- 当前对模糊输出的处理较宽松：若 JSON 解析失败，会退回普通文本 `respond`

## 10. 验收重点

目标验收：

- Planner 输出稳定匹配 `PlanDecision` 结构
- 错误、审批、handoff 都可表达为统一决策
- Reflector 能根据工具结果判定重试、结束或降级
- 不同模型供应商输出经过归一化后可被 Runtime 一致消费

当前实现验收重点：

- 模型输出能稳定被解析为 `StepDecision`
- `respond` 与 `tool_call` 两类决策可被 Runtime 一致消费
- 工具结果返回后，后续模型调用可继续形成最终回答
- 更丰富统一决策类型、Reflector、审批 / handoff 表达能力仍属于后续目标态
