# 规划与推理模块设计

## 1. 模块定位

规划与推理模块负责把上下文转成下一步可执行动作。它不是自由聊天层，而是 Runtime 的结构化决策引擎，输出必须符合 `PlanDecision` 契约。

## 2. 设计目标

- 输出结构化而不是自由文本决策
- 将“思考”和“执行”职责解耦
- 允许模型建议，但最终由 Runtime 裁剪和确认
- 为简单场景和复杂场景提供同构扩展路径

## 3. 模块组成

### 3.1 Planner

职责：

- 判断是否需要工具
- 判断是否需要检索
- 判断是否需要 handoff
- 判断是否可直接回答

### 3.2 Executor

职责：

- 执行工具、检索、handoff 对应动作
- 收集并归一化结果
- 统一错误处理

### 3.3 Reflector

职责：

- 判断当前结果是否足够
- 判断是否需要重试或降级
- 判断是否结束或转人工

## 4. PlanDecision 设计

建议字段：

- `decision_id`
- `type`
- `reason`
- `confidence`
- `selected_model`
- `response_message`
- `tool_calls`
- `retrieval_request`
- `handoff_request`
- `approval_request`
- `terminate_reason`

`type` 枚举建议：

- `respond`
- `tool_call`
- `retrieve`
- `handoff`
- `request_approval`
- `request_human`
- `fail`

## 5. 关键约束

- 任一时刻只允许一个主 `type`
- 多工具调用必须声明是否允许并行
- handoff 必须包含目标 Agent、上下文范围和等待模式
- `fail` 必须返回结构化错误码
- Planner 不得绕过审批或能力边界

## 6. 执行优先级

建议固定如下优先级：

1. 租户级安全策略
2. AgentDefinition 能力边界
3. Runtime 当前状态约束
4. 审批策略
5. Planner 输出建议

如果低优先级与高优先级冲突，由 Runtime 拒绝并生成降级或失败决策。

## 7. 输入输出接口

输入：

- 当前 Run 状态
- 上下文消息
- AgentDefinition
- 可见工具集
- 检索结果
- 已有 Step 与 Tool Result

输出：

- 标准 `PlanDecision`
- 可审计的简短 reason
- 可选的 `selected_model`

## 8. 落地实现建议

- Planner 初期可以与模型调用一一对应
- Reflector 可在 Phase 1 以规则优先实现
- 复杂场景再拆分为独立策略模块
- 所有模型输出都先经 JSON Schema 校验

## 9. 风险控制点

- 不接受“模糊工具意图”作为最终输出
- 不依赖完整链路思维作为恢复依据
- 模型失败时要有 `fail` 或 `request_human` 降级出口
- 并行工具计划需要由 Runtime 再次裁剪

## 10. 验收重点

- Planner 输出稳定匹配 `PlanDecision` 结构
- 错误、审批、handoff 都可表达为统一决策
- Reflector 能根据工具结果判定重试、结束或降级
- 不同模型供应商输出经过归一化后可被 Runtime 一致消费
