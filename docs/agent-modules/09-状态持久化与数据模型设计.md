# 状态持久化与数据模型设计

## 1. 模块定位

状态持久化与数据模型模块负责保存 Agent 平台的事实记录和运行状态，支撑恢复执行、审计追踪、事件投递和查询接口。它要解决“发生了什么”和“下一步该做什么”两个问题，但两类信息需要分层存储。

> 当前实现状态：当前已经落地 `AgentRun`、`AgentStep`、`AgentDefinition`、`AgentDefinitionVersion`、`ToolDefinition` 等核心实体，能够支撑运行快照、步骤查询和 `WaitingTool` 恢复；但更完整的 outbox、审计日志、工具调用实体驱动状态推进等仍未落地。

## 2. 设计目标

- 明确区分审计日志与运行状态
- 为恢复、幂等和回调提供持久化基础
- 支持高频读取当前状态和低频审计回放
- 为事件流提供 outbox 模式支撑

> 当前实现状态：
> - 运行状态与定义数据已初步分层。
> - 当前运行恢复主要依赖 `agent_run + agent_step + wait_token + status_version`。
> - outbox 模式、独立审计日志层、基于工具调用实体的恢复语义仍属于目标态。

## 3. 存储分层

### 3.1 Definition Store

目标对象：

- `agent_definition`
- `agent_definition_version`
- `tool_definition`
- `route_rule`

当前实现状态：

- 当前已实际落地：
  - `agent_definition`
  - `agent_definition_version`
  - `tool_definition`
- `route_rule` 仍处于设计态，尚未进入当前运行时主链路

### 3.2 Runtime Store

目标对象：

- `agent_run`
- `agent_step`
- `agent_message`
- `agent_approval`
- `tool_invocation`
- `idempotency_record`
- `run_outbox_event`

当前实现状态：

- 当前主运行时真正依赖的核心实体是：
  - `agent_run`
  - `agent_step`
- `agent_run` 当前承载：
  - 运行状态 `Status`
  - 当前步骤号 `CurrentStepNo`
  - 并发版本 `StatusVersion`
  - 恢复令牌 `CurrentWaitToken`
  - 幂等键 `IdempotencyKey`
- `agent_step` 当前承载步骤明细，包括：
  - `model_call`
  - `tool_call`
  - `failed`
- `agent_message`、`agent_approval`、`tool_invocation`、`idempotency_record`、`run_outbox_event` 目前仍主要处于目标设计层，尚未形成当前代码主路径依赖

### 3.3 Audit Store

目标对象：

当前实现状态：

- 这意味着当前“步骤记录”同时承担了部分运行追踪职责，但尚不是完整审计层

### 3.5 Memory Store

## 4. 关键数据建模原则

- Run 代表一次执行实例
- Step 代表运行中的一个原子步骤
- Message 代表上下文单元
- ToolInvocation 代表一次具体工具调用
- 审计日志不直接驱动状态推进

> 当前实现状态：
> - 前两条已经落地：`AgentRun` 与 `AgentStep` 是当前状态推进核心。
> - `Message` 还不是当前运行时重建上下文的主对象。
> - `ToolInvocation` 还不是当前状态推进核心对象；当前工具挂起恢复主要依赖 `AgentRun + AgentStep`。

## 5. 关键索引建议

- `agent_run(idempotency_key)` 唯一索引
- `agent_step(run_id, step_key)` 唯一索引
- `tool_invocation(idempotency_key)` 唯一索引
- `agent_run(root_run_id)` 普通索引
- `agent_message(run_id, created_at)` 普通索引
- `run_outbox_event(run_id, seq_no)` 唯一索引

> 当前实现状态：
> - 当前最贴近实际实现的是：
>   - `idempotency_key`
>   - `step_key`
>   - `status_version`
>   - `wait_token`
> - `tool_invocation` 和 `run_outbox_event` 相关索引仍属于未来演进建议。

## 6. 事务边界建议

目标建议纳入同一本地事务：

- 创建或更新 Run
- 创建 Step
- 写入 ToolInvocation
- 写入 Outbox Event

事件实际发送和外部工具调用可在事务外执行，但必须依赖已落库记录重试。

当前实现状态：

- 当前已形成“更新 Run / 写 Step / 后台继续执行”的基本分层
- 但尚未形成 `ToolInvocation + Outbox Event` 一起纳入事务的完整模型
- 当前外部工具调用、事件发布和状态落库之间仍偏 MVP 简化实现

## 7. 恢复语义支撑

目标设计：

- 所有等待状态都要保存 `wait_token`
- 所有关键更新都要携带 `status_version`
- 回调接口必须能通过 `run_id + step_id + wait_token` 定位上下文
- 同一 `step_key` 只能成功一次

当前实现状态：

- 当前已落地：
  - `CurrentWaitToken`
  - `StatusVersion`
  - `StepKey`
- 当前恢复接口主要通过：
  - `run_id`
  - `wait_token`
- 当前还**没有**通过 `step_id + wait_token` 恢复，也没有完整外部回调模型

## 8. 查询模型建议

目标设计：

- 运行快照查询面向 `agent_run`
- 步骤详情查询面向 `agent_step`
- 对话重建查询面向 `agent_message`
- 审计检索面向 `*_log`

当前实现状态：

- 当前已落地：
  - 运行快照查询面向 `agent_run`
  - 步骤详情查询面向 `agent_step`
- `agent_message` 查询视图和独立审计检索尚未形成

## 9. 风险控制点

目标设计：

- 不将审计日志误用为状态源
- 不在未持久化状态时先做外部副作用
- 不让多张表之间出现无来源的孤儿记录
- 不让 outbox 与真实状态脱节

当前实现状态：

- 当前最重要的风险控制点是：
  - 不让 `WaitingTool` 与 `CurrentWaitToken` 脱节
  - 不让 `AgentRun.CurrentStepNo` 与 `AgentStep` 序列脱节
  - 不让 pending step 与 resume 恢复语义不一致
- outbox 一致性问题目前尚未进入当前主链路

## 10. 验收重点

目标验收：

- Runtime 崩溃后可依赖持久化记录恢复
- 事件序列能从 `run_outbox_event` 重放
- 幂等键、Step Key 和 Wait Token 都有唯一性保障
- 查询接口能稳定重建 Run、Step、Message 三类视图

当前实现验收重点：

- `AgentRun` 与 `AgentStep` 能稳定重建当前运行快照与步骤视图
- `WaitingTool + waitToken + resume` 主链路可恢复执行
- `StatusVersion`、`StepKey`、`IdempotencyKey` 已进入当前运行模型
- `run_outbox_event` 重放与 `Message` 视图重建仍属后续目标态
