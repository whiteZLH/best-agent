# 状态持久化与数据模型设计

## 1. 模块定位

状态持久化与数据模型模块负责保存 Agent 平台的事实记录和运行状态，支撑恢复执行、审计追踪、事件投递和查询接口。它要解决“发生了什么”和“下一步该做什么”两个问题，但两类信息需要分层存储。

## 2. 设计目标

- 明确区分审计日志与运行状态
- 为恢复、幂等和回调提供持久化基础
- 支持高频读取当前状态和低频审计回放
- 为事件流提供 outbox 模式支撑

## 3. 存储分层

### 3.1 Definition Store

- `agent_definition`
- `agent_definition_version`
- `tool_definition`
- `route_rule`

### 3.2 Runtime Store

- `agent_run`
- `agent_step`
- `agent_message`
- `agent_approval`
- `tool_invocation`
- `idempotency_record`
- `run_outbox_event`

### 3.3 Audit Store

- `model_call_log`
- `tool_execution_log`
- `policy_audit_log`

### 3.4 Knowledge Store

- `knowledge_document`
- `knowledge_chunk`
- `embedding_index`

### 3.5 Memory Store

- `session_memory`
- `user_memory`
- `summary_memory`

## 4. 关键数据建模原则

- Run 代表一次执行实例
- Step 代表运行中的一个原子步骤
- Message 代表上下文单元
- ToolInvocation 代表一次具体工具调用
- 审计日志不直接驱动状态推进

## 5. 关键索引建议

- `agent_run(idempotency_key)` 唯一索引
- `agent_step(run_id, step_key)` 唯一索引
- `tool_invocation(idempotency_key)` 唯一索引
- `agent_run(root_run_id)` 普通索引
- `agent_message(run_id, created_at)` 普通索引
- `run_outbox_event(run_id, seq_no)` 唯一索引

## 6. 事务边界建议

推荐纳入同一本地事务：

- 创建或更新 Run
- 创建 Step
- 写入 ToolInvocation
- 写入 Outbox Event

事件实际发送和外部工具调用可在事务外执行，但必须依赖已落库记录重试。

## 7. 恢复语义支撑

- 所有等待状态都要保存 `wait_token`
- 所有关键更新都要携带 `status_version`
- 回调接口必须能通过 `run_id + step_id + wait_token` 定位上下文
- 同一 `step_key` 只能成功一次

## 8. 查询模型建议

- 运行快照查询面向 `agent_run`
- 步骤详情查询面向 `agent_step`
- 对话重建查询面向 `agent_message`
- 审计检索面向 `*_log`

## 9. 风险控制点

- 不将审计日志误用为状态源
- 不在未持久化状态时先做外部副作用
- 不让多张表之间出现无来源的孤儿记录
- 不让 outbox 与真实状态脱节

## 10. 验收重点

- Runtime 崩溃后可依赖持久化记录恢复
- 事件序列能从 `run_outbox_event` 重放
- 幂等键、Step Key 和 Wait Token 都有唯一性保障
- 查询接口能稳定重建 Run、Step、Message 三类视图
