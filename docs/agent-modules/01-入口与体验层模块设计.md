# 入口与体验层模块设计

## 1. 模块定位

入口与体验层是 Agent 平台的用户接入面，承载 Chat UI、Copilot UI、Workflow UI、Approval UI 和 Admin Console。它的目标不是实现智能决策，而是把运行过程以可交互、可追踪、可审批的方式呈现给用户和运营人员。

## 2. 设计目标

- 为不同使用场景提供统一但可扩展的交互入口
- 屏蔽底层 Runtime、Model、Tool 的实现细节
- 支持流式输出、步骤状态展示、审批介入和人工接管
- 支持多租户、多环境、多角色视图隔离

## 3. 职责边界

负责：

- 用户输入采集
- 流式消息展示
- Step 级状态回显
- 审批任务展示与处理入口
- 运行记录查询和管理控制台

不负责：

- 运行状态推进
- 工具权限判断
- 模型调用编排
- 业务副作用执行

## 4. 核心页面与能力

### 4.1 Chat UI

- 展示多轮对话
- 展示工具调用和检索摘要
- 支持流式增量消息拼接
- 支持中断、取消、恢复

### 4.2 Copilot UI

- 面向嵌入式场景
- 支持上下文感知的辅助提问
- 支持宿主系统透传身份和会话信息

### 4.3 Workflow UI

- 展示长任务进度
- 展示当前状态、等待原因、最近 Step
- 支持异步任务回看和恢复

### 4.4 Approval UI

- 展示待审批动作、风险等级和请求载荷
- 支持批准、拒绝、转人工
- 展示审批过期时间和审批审计记录

### 4.5 Admin Console

- 管理 Agent 定义和版本
- 查看运行统计、错误日志、成本和策略命中情况
- 配置工具 allowlist、审批规则和知识源

## 5. 交互模型

- 所有入口都统一基于 `runId` 追踪一次执行
- 前端以 `run snapshot + event stream` 作为核心数据模型
- 对话视图消费 `message.delta`
- 步骤视图消费 `step.started`、`step.completed`
- 审批视图消费 `approval.requested`、`approval.resolved`

## 6. 核心接口依赖

- `POST /agent-runs`
- `GET /agent-runs/{runId}`
- `GET /agent-runs/{runId}/steps`
- `GET /agent-runs/{runId}/events?afterSeqNo=...`
- `POST /agent-runs/{runId}:cancel`
- `POST /agent-runs/{runId}:resume`
- `POST /agent-runs/{runId}/steps/{stepId}:approve`

## 7. 关键设计约束

- UI 不直接拼接业务状态，统一使用结构化事件和状态枚举
- 前端必须基于 `seqNo` 做去重、补洞和重连恢复
- 审批和人工接管必须具备权限校验，不允许仅靠前端按钮控制
- 长任务场景必须支持断线后重建界面状态

## 8. 失败与降级

- 事件流断开时退回轮询 `GET /agent-runs/{runId}`
- 流式消息缺片时按 `seqNo` 补拉事件
- 审批 UI 过期后只允许查看，不允许再次提交
- 如果模型推理不可见，至少保证 Step 状态可见

## 9. 验收重点

- 同一 Run 可在 Chat UI 和 Admin Console 中一致展示
- 流式输出中断后可从最后 `seqNo` 恢复
- 审批任务可准确回显触发原因、风险等级和结果
- 人工接管、取消、恢复动作都有明确权限与审计
