# 审批与人机协同模块设计

## 1. 模块定位

审批与人机协同模块负责在高风险、不确定或需要人为确认的场景下中断自动执行，并让人介入判断、补充信息、替换结果或终止任务。

## 2. 设计目标

- 对高风险动作建立可审计闸门
- 将审批等待纳入 Runtime 的正式状态机
- 支持审批、转人工和人工接管三类模式
- 保证等待、超时、恢复和回调幂等

## 3. 审批触发条件

建议支持：

- 特定工具
- 特定参数模式
- 特定租户策略
- 特定风险等级

## 4. 核心对象

### 4.1 ApprovalDecision

建议字段：

- `decision`
- `approver_id`
- `approver_role`
- `comment`
- `decided_at`

### 4.2 ApprovalRequest

建议至少包含：

- `approval_id`
- `run_id`
- `step_id`
- `requested_action`
- `risk_level`
- `request_payload`
- `expires_at`
- `wait_token`

## 5. 流程设计

### 5.1 审批流程

1. Runtime 命中审批策略
2. 创建 `approval_request` Step
3. Run 状态切换为 `WaitingApproval`
4. 外部审批系统或人工台处理
5. 写入 `approval_result`
6. 恢复 Run

### 5.2 人工接管流程

允许人工：

- 补充信息
- 替换工具结果
- 终止任务

人工接管后仍应形成结构化 Step 和审计日志。

## 6. 超时与默认策略

必须明确：

- 审批超时后默认是拒绝、取消还是转人工
- 过期审批结果是否允许回写
- 审批恢复前是否需要重新校验上下文变化

## 7. 权限与审计

- 审批操作必须有身份鉴权
- 审批动作必须记录审批人和角色
- 人工替换工具结果必须保留原结果与替换原因
- 所有接管动作都要关联 `run_id` 和 `step_id`

## 8. 风险控制点

- 防止前端直接绕过审批接口
- 防止过期结果恢复已变化的 Run
- 防止重复审批导致多次恢复
- 防止人工接管覆盖原始审计链路

## 9. 验收重点

- 高风险工具可稳定触发 `WaitingApproval`
- 审批结果回写具备幂等性
- 人工补充和人工终止可被 Runtime 正确消费
- 超时场景具备明确且可测试的默认行为
