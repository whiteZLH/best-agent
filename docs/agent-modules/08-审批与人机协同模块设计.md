# 审批与人机协同模块设计

## 1. 模块定位

审批与人机协同模块负责在高风险、不确定或需要人为确认的场景下中断自动执行，并让人介入判断、补充信息、替换结果或终止任务。

> 当前实现状态：该模块目前基本仍处于目标设计阶段。当前运行时只支持 `WaitingTool` 异步工具挂起与恢复，尚未实现 `WaitingApproval`、人工接管、审批回调和人工替换结果主链路。

## 2. 设计目标

- 对高风险动作建立可审计闸门
- 将审批等待纳入 Runtime 的正式状态机
- 支持审批、转人工和人工接管三类模式
- 保证等待、超时、恢复和回调幂等

> 当前实现状态：
> - 当前尚未把审批等待纳入正式状态机。
> - 当前也未形成独立的人机协同接口或审批恢复语义。
> - 当前与人工介入最接近的能力，是工具挂起后的 `POST /agent-runs/{runId}:resume`，但它面向的是异步工具恢复，不是审批链路。

## 3. 审批触发条件

目标建议支持：

- 特定工具
- 特定参数模式
- 特定租户策略
- 特定风险等级

当前实现状态：

- 当前尚未实现审批触发条件解析
- `ToolDefinition` 中已经存在 `side_effect_level` 等元数据，可作为未来审批策略输入
- 但当前运行时尚未根据这些信息触发 `WaitingApproval`

## 4. 核心对象

### 4.1 ApprovalDecision

目标字段：

- `decision`
- `approver_id`
- `approver_role`
- `comment`
- `decided_at`

当前实现状态：

- 尚未形成当前代码中的审批决策对象

### 4.2 ApprovalRequest

目标至少包含：

- `approval_id`
- `run_id`
- `step_id`
- `requested_action`
- `risk_level`
- `request_payload`
- `expires_at`
- `wait_token`

当前实现状态：

- 相关表结构和对象在总体设计中已有方向
- 但当前代码主链路尚未真正创建审批请求对象，也未将其纳入运行恢复模型

## 5. 流程设计

### 5.1 审批流程

目标流程：

1. Runtime 命中审批策略
2. 创建 `approval_request` Step
3. Run 状态切换为 `WaitingApproval`
4. 外部审批系统或人工台处理
5. 写入 `approval_result`
6. 恢复 Run

当前实现状态：

- 当前尚未落地该流程
- 当前运行时正式支持的等待状态只有 `WaitingTool`
- 当前也没有审批结果回写后的恢复路径

### 5.2 人工接管流程

目标能力：允许人工：

- 补充信息
- 替换工具结果
- 终止任务

人工接管后仍应形成结构化 Step 和审计日志。

当前实现状态：

- 当前尚未落地人工接管流程
- 当前也没有“人工替换工具结果”或“人工终止任务”的独立运行时动作模型

## 6. 超时与默认策略

目标设计必须明确：

- 审批超时后默认是拒绝、取消还是转人工
- 过期审批结果是否允许回写
- 审批恢复前是否需要重新校验上下文变化

当前实现状态：

- 尚未落地
- 当前只有异步工具超时与 webhook 超时处理，不涉及审批超时策略

## 7. 权限与审计

目标设计：

- 审批操作必须有身份鉴权
- 审批动作必须记录审批人和角色
- 人工替换工具结果必须保留原结果与替换原因
- 所有接管动作都要关联 `run_id` 和 `step_id`

当前实现状态：

- 当前尚未落地审批身份鉴权与审批审计链路
- 当前运行审计更多依赖 `AgentStep` 与事件流，而不是审批专用记录

## 8. 风险控制点

目标设计：

- 防止前端直接绕过审批接口
- 防止过期结果恢复已变化的 Run
- 防止重复审批导致多次恢复
- 防止人工接管覆盖原始审计链路

当前实现状态：

- 这些风险控制点目前大多尚未进入当前主链路
- 当前最接近的已实现保护是：`WaitingTool` 恢复时对 `waitToken` 和 `run.Status` 的校验
- 它可以视为未来审批恢复幂等校验的参考模式，但不能等价替代审批机制

## 9. 验收重点

目标验收：

- 高风险工具可稳定触发 `WaitingApproval`
- 审批结果回写具备幂等性
- 人工补充和人工终止可被 Runtime 正确消费
- 超时场景具备明确且可测试的默认行为

当前实现验收重点：

- 当前该模块尚未进入可验收落地状态
- 后续真正实现时，建议优先复用当前 `WaitingTool + waitToken + resume` 已验证过的挂起/恢复模式，再扩展到审批专用状态与接口
