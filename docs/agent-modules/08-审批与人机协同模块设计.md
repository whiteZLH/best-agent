# 审批与人机协同模块设计

## 1. 模块定位

审批与人机协同模块负责在高风险、不确定或需要人为确认的场景下中断自动执行，并让人介入判断、补充信息、替换结果或终止任务。

> 当前实现状态：当前已经落地最小审批闭环：高风险工具可触发 `WaitingApproval`，支持 approve / reject 恢复路径，也支持外部审批系统通过 `approvalId` complete 回调回写决策，并通过 `AgentApproval` 独立记录审批请求、审批人、决策和审计时间。最小人工接管闭环也已落地，并已支持在 `WaitingTool` 场景下用人工结果替代挂起工具回调继续运行；当前也已开始支持对已完成 `tool_call` 步骤发起人工覆盖，并在保留原工具结果审计信息的同时用人工结果继续生成最终答复；同时已补上“仅允许覆盖当前 run 最新已完成且仍为当前步骤的 `tool_call` 结果”的最小服务端保护，并要求 `request-human` / `complete-human` 具备明确人工操作者身份；审批超时默认拒绝策略与最小外部回调来源认证也已落地。

## 2. 设计目标

- 对高风险动作建立可审计闸门
- 将审批等待纳入 Runtime 的正式状态机
- 支持审批、转人工和人工接管三类模式
- 保证等待、超时、恢复和回调幂等

> 当前实现状态：
> - 当前已经把 `WaitingApproval` 纳入运行时状态机。
> - 当前审批恢复接口为 `POST /agent-runs/{runId}/steps/{stepId}:approve` 与 `POST /agent-runs/{runId}/steps/{stepId}:reject`。
> - 当前外部审批回调接口为 `POST /agent-runs/{runId}/approvals/{approvalId}:complete`。
> - 当前已落地最小人机协同接口：`request-human` 和 `complete-human`。
> - 当前已支持“替代挂起工具结果”的最小人工替换模型。
> - 当前已支持对已完成 `tool_call` 结果发起人工覆盖，并保留原结果审计信息。
> - 当前已落地审批超时默认拒绝策略。
> - 当前已落地最小外部回调来源认证：可对 tool complete / approval complete 进行 HMAC 签名校验。

## 3. 审批触发条件

目标建议支持：

- 特定工具
- 特定参数模式
- 特定租户策略
- 特定风险等级

当前实现状态：

- 当前运行时已经根据 `ToolDefinition.SideEffectLevel` 触发审批等待
- `internal_write`、`external_write`、`destructive` 当前默认会触发 `WaitingApproval`
- 当前已支持通过 `Approval:Policy` 配置审批风险级别与审批角色规则
- 当前也已支持由 `AgentDefinitionVersion.ApprovalPolicy` 在单个 Agent 版本范围内覆盖全局默认审批策略；run 启动后会跟随已绑定版本生效，不受后续激活新版本影响；在 handoff 子链路中，子 Agent 的有效审批策略当前也已开始按父子更严格的一侧收紧
- 当前已支持最小参数模式审批策略：可按 `Approval:Policy:ParameterApprovalRules` 基于 `toolName + inputPath + expectedValue` 触发审批，并可选覆盖审批风险等级
- 当前审批策略配置已开始显式规范化：`ApprovalRequiredSideEffectLevels`、`RoleRequiredSideEffectLevels` 与 `ParameterApprovalRules.OverrideSideEffectLevel` 会统一收紧为合法 side effect level 枚举，并在应用注册阶段 fail-fast 拒绝无效配置
- 当前也已开始支持最小租户级审批策略解析：可通过 `Approval:TenantPolicies` 按 `tenantId` 配置审批触发等级、审批角色和参数规则；Worker 的审批触发判断以及 step 级 `approve/reject` / 外部 `approval complete` 都会按当前 run 的 `tenantId` 解析租户策略，并与版本策略继续收敛到更严格结果

## 4. 核心对象

### 4.1 ApprovalDecision

目标字段：

- `decision`
- `approver_id`
- `approver_role`
- `comment`
- `decided_at`

当前实现状态：

- 当前 `AgentApproval` 持久化审批决策字段
- `AgentStep.DecisionPayload` 中也保留运行时审批快照，用于兼容和恢复

### 4.2 ApprovalRequest

目标至少包含：

- `approval_id`
- `run_id`
- `step_id`
- `requested_action`
- `risk_level`
- `request_payload`
- `expires_at`
- `wait_token`

当前实现状态：

- 当前代码主链路会创建独立 `AgentApproval` 记录
- 审批请求关联 `run_id`、`step_id`、`requested_action`、`risk_level`、`request_payload` 和 `wait_token`
- 当前主链路会为待审批记录写入 `expires_at`
- 后台超时调度器会消费过期的 pending 审批

## 5. 流程设计

### 5.1 审批流程

目标流程：

1. Runtime 命中审批策略
2. 创建 `approval_request` Step
3. Run 状态切换为 `WaitingApproval`
4. 外部审批系统或人工台处理
5. 写入 `approval_result`
6. 恢复 Run

当前实现状态：

- 当前已落地最小审批流程
- Runtime 命中写类风险工具后会创建 Pending `tool_call` step，并由 Worker 创建 `AgentApproval`
- Planner 显式返回 `request_approval` 时，也会创建 Pending `approval_request` step 并复用同一条审批闭环；approve 后会把审批结果喂回下一轮规划，reject 则终止 Run
- Run 会切换为 `WaitingApproval`
- approve 会执行待批工具并恢复 loop；reject 会终止 Run
- 当前支持 step 级 approve / reject API，也支持外部审批系统按 `approvalId` complete 回调回写 `Approved` / `Rejected`
- 当前外部审批 complete 回调支持 HTTP `Idempotency-Key`，同 key 同 payload 会重放已记录结果，不重复入队；同 key 不同 payload 会被拒绝
- 当前外部审批 complete 回调已支持最小来源认证：可按配置要求 HMAC 签名
- 当前审批超时会由后台 `ApprovalTimeoutDispatcher` 扫描过期 pending 审批，并按默认拒绝策略终止 Run

### 5.2 人工接管流程

目标能力：允许人工：

- 补充信息
- 替换工具结果
- 终止任务

人工接管后仍应形成结构化 Step 和审计日志。

当前实现状态：

- 当前已落地最小人工接管流程：
  - `POST /agent-runs/{runId}:request-human` 可将 `Running` / `WaitingTool` / `WaitingApproval` 的 Run 切换到 `WaitingHuman`
  - Runtime 会创建 Pending `human_wait` step，并写入 `waiting_human` 事件
  - `POST /agent-runs/{runId}/steps/{stepId}:complete-human` 会校验 `waitToken`、当前 pending step 和人工等待 payload
  - 若 `terminate = false` 且来源是普通接管，人工结果会补全 `human_wait` step，并直接完成 Run
  - 若 `terminate = false` 且来源是 `WaitingTool`，人工结果会作为替代工具结果继续 Runtime loop
  - 若 `terminate = true`，人工终止会将 `human_wait` step 与 Run 标记为失败
- 当前 `AgentStep.DecisionPayload` 已记录人工处理人、角色、评论、处理时间和人工结果，并可记录来源 `tool_wait` / `approval_wait` / `run` 上下文
- 当前 `WaitingTool -> request-human -> complete-human` 已支持最小人工替换工具结果模型：原 pending `tool_call` 与 `ToolInvocation` 会被标记为 `Cancelled`，`human_wait` step 会保留来源工具名、输入和替换原因，再由人工结果继续 loop
- 当前 `Running -> request-human(sourceStepId=已完成tool_call) -> complete-human` 也已支持更通用的人工覆盖结果模型：`human_wait` step 会保留来源 `stepId`、`invocationId`、原工具输入、原工具输出和原状态，再由人工结果继续生成最终答复
- 当前工具失败补偿也已开始复用同一条人工接管链路：当工具声明 `CompensationPolicy={"mode":"manual"}` 且执行失败、返回 `failed/error`，或替代结果命中输出 schema 校验失败时，运行时会自动创建 `sourceType = tool_failure` 的 `human_wait` step，要求人工补偿后再继续
- 当前对已完成工具结果的人工覆盖已增加最小服务端保护：只允许接管当前 run 中最新一条已完成、且仍为当前步骤并有输出的 `tool_call`，避免在运行已经继续推进后回溯改写更早的工具结果

## 6. 超时与默认策略

目标设计必须明确：

- 审批超时后默认是拒绝、取消还是转人工
- 过期审批结果是否允许回写
- 审批恢复前是否需要重新校验上下文变化

当前实现状态：

- 当前已落地最小审批超时策略：
  - `AgentApproval.ExpiresAt` 在进入 `WaitingApproval` 时写入
  - 后台 `ApprovalTimeoutDispatcher` 会按配置扫描过期 pending 审批
  - 默认策略为拒绝：过期后将 `AgentApproval` 标记为 `Rejected`，将 pending step 标记为 `Failed`，并将 Run 推进到 `TimedOut` 终态
  - 当前也支持按 `Approval:TimeoutAction=request_human` 转人工：过期审批会先按超时拒绝收尾，再创建 `human_wait` step 并把 Run 推进到 `WaitingHuman`
  - 默认拒绝时会同时发布 `approval_timed_out` 与 `error` 事件，事件 `runStatus` / payload `status` 也会同步标记为 `TimedOut`；转人工时则会发布 `approval_timed_out` 与 `waiting_human`
- 更复杂的租户级默认策略仍未落地

## 7. 权限与审计

目标设计：

- 审批操作必须有身份鉴权
- 审批动作必须记录审批人和角色
- 人工替换工具结果必须保留原结果与替换原因
- 所有接管动作都要关联 `run_id` 和 `step_id`

当前实现状态：

- 当前 API 会优先从 `HttpContext.User` 解析审批人身份，无认证用户时兼容回退请求体字段
- 当前人工接管接口也会优先从 `HttpContext.User` 解析人工处理人身份；若无认证用户，则仍要求请求体显式提供人工处理人身份，否则会在应用层拒绝处理
- 当前应用层通过 `IApprovalAuthorizer` 执行最小审批授权：审批人身份必填，写类风险默认要求 `approver`、`reviewer`、`admin` 或 `owner` 角色；若 run 绑定版本声明了 `ApprovalPolicy`，则 step 级 `approve/reject` 与外部 `approval complete` 也会优先按该版本策略做角色校验
- 当前应用层通过 `IHumanTakeoverAuthorizer` 执行最小人工接管授权：人工处理人身份必填；若配置 `HumanTakeover:AllowedRoles`，则 `request-human` / `complete-human` 还会要求人工处理人角色命中允许集合
- 当前审批触发规则和审批角色规则都已提升为可配置策略：
  - `Approval:Policy:ApprovalRequiredSideEffectLevels`
  - `Approval:Policy:RoleRequiredSideEffectLevels`
  - `Approval:Policy:AllowedApproverRoles`
- 当前租户级审批策略也已开始支持最小配置：
  - `Approval:TenantPolicies`
- 当前人工接管角色规则也已开始支持最小配置：
  - `HumanTakeover:AllowedRoles`
- 当前 step 级审批接口和外部 approval complete 回调都会复用同一套审批授权规则
- 当前 tool complete / approval complete 外部回调可按 `WebhookSecurity` 配置要求携带 HMAC 签名头；签名失败会在 API 边界直接返回 401
- 当前 `AgentApproval` 会记录审批人、角色、评论、决策时间与审计修改人
- 当前人工接管接口也会把人工处理人、角色、评论和人工结果写入 `human_wait` step 的决策 payload
- 当前挂起工具结果的人为替代也会把来源 `stepId`、`invocationId`、工具名、工具输入和“是否继续作为工具结果”写入 `human_wait` step 的决策 payload
- 当前已完成工具结果的人为覆盖也会把原工具输出与原状态一并写入 `human_wait` step 的决策 payload
- 当前虽然已开始支持最小租户级审批策略解析，但仍未接入更完整的正式认证鉴权中间件、租户隔离或更细粒度的可配置权限策略

## 8. 风险控制点

目标设计：

- 防止前端直接绕过审批接口
- 防止过期结果恢复已变化的 Run
- 防止重复审批导致多次恢复
- 防止人工接管覆盖原始审计链路

当前实现状态：

- 当前 approve / reject / approval complete 会校验 Run 状态、当前 Pending step、审批 payload 决策状态和审批授权规则
- 外部 approval complete 会额外通过 `approvalId` 查询 `AgentApproval`，并确认它仍是当前 pending 审批
- 外部 approval complete 会使用 `IdempotencyRecord` 记录同一 `Idempotency-Key` 的完成结果，避免重复回调二次恢复
- 当前 complete-human 会校验 `waitToken`、当前 pending `human_wait` step 和人工等待 payload，避免误恢复错误 Run
- 当前 request-human 从 `WaitingTool` / `WaitingApproval` 进入时，会同步收尾原 pending `ToolInvocation` / `AgentApproval`，避免状态分叉
- 当前审批超时处理也会校验 Run 仍处于 `WaitingApproval`、`waitToken` 仍匹配且当前 step 仍是 pending，避免对已恢复 Run 重复终止
- `StatusVersion` 作为 EF Core 并发令牌，用于降低并发审批或重复恢复风险
- 当前已实现最小外部回调来源认证
- 当前已补上“仅允许覆盖当前 run 最新已完成且仍为当前步骤的 `tool_call` 结果”的最小人工接管覆盖保护
- 当前已补上人工接管最小身份闸门：`request-human` / `complete-human` 都要求明确人工操作者身份；若配置 `HumanTakeover:AllowedRoles`，还会进一步拒绝不在允许角色集合内的人工操作者
- 当前仍未实现更细粒度权限边界、正式认证鉴权以及更丰富的覆盖前置策略

## 9. 验收重点

目标验收：

- 高风险工具可稳定触发 `WaitingApproval`
- 审批结果回写具备幂等性
- 人工补充和人工终止可被 Runtime 正确消费
- 超时场景具备明确且可测试的默认行为

当前实现验收重点：

- 已可验收最小审批闭环：高风险工具触发 `WaitingApproval`、approve / reject 恢复、外部 approval complete 回调与幂等重放、`AgentApproval` 审计、审批授权拒绝
- 已可验收最小审批超时闭环：`expires_at` 写入、后台过期扫描、默认拒绝并进入 `TimedOut` 终态、事件发布
- 已可验收最小外部回调来源认证闭环：tool complete / approval complete 可按 HMAC 签名校验来源，校验失败返回 401
- 已可验收最小人工接管闭环：`WaitingHuman`、人工补充结果完成 Run、人工终止失败 Run、`human_wait` 审计 payload 与事件流
- 已可验收最小人工替换挂起工具结果闭环：`WaitingTool -> request-human -> complete-human` 可继续 loop，且保留来源工具上下文与替换原因
- 已可验收已完成工具结果的人为覆盖闭环：`Running -> request-human(sourceStepId) -> complete-human` 可保留原工具结果审计信息并继续生成最终答复
- 下一阶段验收重点应转向：正式认证鉴权与租户权限、更细粒度审批策略，以及更丰富的人工接管覆盖前置策略
